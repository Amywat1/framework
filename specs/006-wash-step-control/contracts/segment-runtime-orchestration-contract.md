# 契约：工步段运行编排

## 目标

定义应用层如何推进工步段生命周期，以及 stop/fault/timeout/exit_failure 如何统一收口。

## 生命周期契约

| 生命周期阶段 | 应用层职责 | 领域层职责 | 平台层职责 |
|------|------|------|------|
| 进入段 | 装载当前工步、初始化运行视图、发起必要启动动作 | 校验本段可进入条件 | 执行启动命令并提供反馈 |
| 运行段 | 持续采集位置与反馈、评估条件控制启停、判断本段完成 | 依据强类型规则判定是否应启停控制与完成本段 | 提供位置/反馈事实，执行持续控制输出 |
| 退出段 | 启动退出动作、等待关闭/回零完成、处理退出超时 | 判定退出动作顺序与完成规则 | 执行停刷/停药/回零/关闭输出并提供反馈 |
| 完成/中止 | 统一写入结果与状态投影 | 给出段结果与异常决议 | 进入最终停机或下一段准备状态 |

## 异常收口契约

| 事件 | 统一收口要求 |
|------|------|
| `fault` | 应用层决定立即中止还是先执行安全收尾 |
| `stop` | 应用层进入停机或收尾路径，不允许旁路写状态 |
| 段内超时 | 必须生成“工步未完成”类结果 |
| 收尾超时 | 必须生成“退出未完成”类结果 |
| 跟随失效 | 必须按异常策略进入中止或安全收尾 |
| 药剂关闭失败 | 必须进入退出失败语义，不得视为正常完成 |

## 编排约束

1. 应用层负责“怎么调用”，但不负责解释 JSON 文本字段。
2. 领域层负责工艺规则，但不直接写最终结果投影。
3. 平台层不解释工步段语义，只提供事实和执行命令。
4. 若旧阶段推进路径与此契约冲突，旧路径必须废弃。

## 验证准则

- 任一工步段都能按进入、运行、退出三段式持续推进。
- 顶刷回零、RO 水关闭、风干关闭都在退出段中正式完成。
- `fault / stop / timeout / exit_failure` 都通过同一编排入口收口。
