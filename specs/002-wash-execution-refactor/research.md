# 研究记录：洗车执行模型重构

## 决策 1：采用“洗车会话 + 单次执行”双层运行时模型

**Decision**：将一次完整洗车任务建模为 `wash_session`，将其内部某一阶段或某一步骤的推进建模为 `wash_execution`。

**Rationale**：
- 规格要求会话状态与执行状态清晰分离。
- 该模型天然支持“执行结束后由会话决定继续、完成或中止”。
- 它能把运行生命周期、推进细节、结果归档和中止原因拆开，提升可观测性和扩展性。

**Alternatives considered**：
- 单层 `wash_cycle` 状态机：结构更简单，但无法清楚表达执行结果回传与会话后续决策。
- 为每个步骤都建立独立对象：灵活度更高，但超出当前最小可行范围。

## 决策 2：业务推进只由事件或等待超时触发

**Decision**：主流程状态迁移只由启动、停止、设备反馈、故障、业务事件或等待超时触发；固定周期检查仅用于“发现事件”或“发现超时”。

**Rationale**：
- 这直接满足规格中对推进方式的刚性约束。
- 可以保留现有 Linux 主循环能力，同时避免把业务状态推进重新退化成轮询逻辑。
- 有利于将平台层“发现事实”和领域层“解释事实”分离。

**Alternatives considered**：
- 让轮询周期直接推进业务状态：实现最短，但违反已澄清规则。
- 完全无周期机制、只依赖外部唤醒：难以稳定发现等待超时。

## 决策 3：程序数据在启动前校验并在会话内冻结为快照

**Decision**：启动事件到来时，先校验程序数据可用性和有效性；只有校验通过才创建会话，并在会话内绑定一份只读程序快照。

**Rationale**：
- 规格已明确启动前校验失败必须拒绝创建会话。
- 会话内快照能保证运行中程序修改不会影响当前任务。
- 复用现有程序仓储即可支持“读取当前版本 + 生成本次快照”，无需引入数据库。

**Alternatives considered**：
- 会话创建后再校验：会引入无效会话和额外终止路径。
- 运行中按需重新读取程序配置：会破坏任务一致性。

## 决策 4：竞争事件采用单一优先级收敛

**Decision**：同一执行窗口内的竞争输入统一按“故障 > 停止 > 超时 > 设备反馈”收敛；迟到事件和重复事件始终忽略。

**Rationale**：
- 该规则已在 clarify 中确认。
- 它能确保最终结果唯一、日志一致、状态迁移可验证。
- 以安全为先的顺序符合控制系统风险控制原则。

**Alternatives considered**：
- 仅按到达顺序处理：简单，但会让并发边界和最终结果依赖时序抖动。
- 停止优先于故障：不符合保守安全策略。

## 决策 5：重复启动请求在运行中直接拒绝，不排队

**Decision**：当存在运行中的洗车会话时，新的启动事件直接拒绝，不排队、不覆盖、不重置当前会话。

**Rationale**：
- 当前系统是单机单车位，排队能力不在规格范围内。
- 拒绝比排队更容易验证，也更符合“范围最小化”的原则。
- 保持当前会话稳定，避免引入隐式队列和后续调度规则。

**Alternatives considered**：
- 允许排队待执行：需要新增队列模型、持久化和取消规则。
- 允许新启动覆盖旧会话：会破坏运行时一致性和安全处理路径。

## 决策 6：会话结束、启动拒绝与忽略事件都需要统一记录

**Decision**：不仅正常终态需要记录，启动拒绝、重复启动拒绝、迟到事件忽略、重复事件忽略和关键超时收敛也都要进入统一的状态迁移或事件记录体系。

**Rationale**：
- 规格要求关键结果可判断、可观测、可验证。
- 仅记录成功和故障不足以支持后续排障和需求验收。
- 统一记录模型能减少测试和实现中的特例分支。

**Alternatives considered**：
- 只记录终态结果：实现更少，但无法支撑拒绝路径和忽略路径验证。
- 为每类拒绝建立单独日志体系：会增加实现复杂度，收益不足。
