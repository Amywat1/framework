# 实施计划：system_context 组合根生命周期收口与静态池化
**分支**：`009-system-context-pooling` | **日期**：2026-05-09 | **规格**：[spec.md](./spec.md)  
**输入**：来自 `specs/009-system-context-pooling/spec.md` 的功能规格

**说明**：本计划围绕“`system_context_t` 作为对外不透明、只能通过正式生命周期获取、且实例唯一来源为内部静态对象池的组合根资源”展开。目标是统一生命周期语义、收回对象存储拥有权、明确固定容量池耗尽行为，并让 `main`、调度器、适配层和测试全部迁移到同一套正式入口，不改变洗车业务流程本身。

## 概要

本次特性把 `system_context_t` 从“既可由库创建、又可由外部自带存储初始化”的混合模型，收口为一套单一正式生命周期。公开头文件只保留不透明声明和正式获取、重置、释放、装配接口；真实结构体、池管理状态和池容量常量留在私有实现侧。内部资源模型固定为**静态对象池**，不允许运行期扩容，也不再保留堆分配和外部自带存储路径。

实现重点不是把堆改成静态区，而是把拥有权和生命周期边界彻底收回。外围模块今后只能“申请句柄 -> 装配 -> 使用 -> 重置/释放”，不能再决定对象存储来源。回归验证也要从“还能不能直接构造结构体”转成“是否守住正式边界、池耗尽行为和异常语义”。

## 技术上下文

**语言/版本**：C17  
**主要依赖**：项目现有 C 代码、公开头文件/私有头文件分层、内部固定容量静态对象池、CMake、CTest  
**存储**：进程内运行态；`system_context` 私有静态对象池；现有文件配置与运行日志路径继续保留  
**测试方式**：CTest 驱动的 `unit` / `contract` / `integration` / `simulation` 测试  
**目标平台**：Linux 仿真与当前单进程主控运行环境  
**项目类型**：单项目 C 静态库 + 主程序  
**性能目标**：正式生命周期获取/释放保持有界且可预测；不引入新的动态分配抖动；不改变现有控制周期语义  
**约束**：不新增语言、不引入第三方对象池库、不引入额外分配器、不扩展线程模型、不新增脚本运行时；公开边界必须保持不透明；内部资源模型必须是固定容量静态对象池；池容量通过私有编译期常量定义，运行期不得扩容  
**规模/范围**：核心修改集中在 `application/coordinators`、`main`、`platform`、`adapters`、`tests` 及相关 CMake 依赖

## 假设、问题与权衡

**假设**

- 当前生产路径在单进程内同时存活的正式 `system_context` 实例数量很小，适合由私有固定容量静态池承载。
- 调度器、适配层和测试路径可以接受从旧的 `create()/initialize()/destroy()` 语义迁移到新的正式生命周期入口。
- 当前大多数只读查询接口仍可继续基于 `system_context_t *` 句柄使用，不必在本期把所有只读 API 全部重写成值句柄风格。

**未决问题**

- 无。当前方向已足够进入设计与实现。

**被拒绝的方案**

- 保留 `system_context_create()` 与 `system_context_initialize()` 双轨入口：会继续放大拥有权歧义，无法真正收回外部自带存储。
- 继续使用 `malloc/free + owns_storage`：虽然比外部自带存储统一一些，但仍保留不稳定的运行时分配路径，不符合本次收口目标。
- 引入第三方对象池库或自定义分配器：超出当前最小可行改造范围，也不符合“不新增技术栈”的约束。
- 改用槽位扫描、并行来源或让 `in_use` 同时承担分配查找职责：被拒绝，因为当前规格、研究和澄清已经固定为“空闲链表头弹出/头插回收 + `in_use` 仅做合法性校验”的单一路径，偏离它会重新引入资源来源与合法性检查耦合。
- 顺手把 `system_context_t` 扩成更通用的上下文容器：会扩大职责面，与“收口边界”的目标冲突。

**最小可行方案**

- 在 `include/application/coordinators/system_context.h` 中移除“调用方自带存储初始化”的公开语义，只保留正式获取、重置、释放和端口装配接口。
- 在 `src/application/coordinators/` 中引入私有固定容量静态对象池，容量由私有编译期常量定义；所有正式实例只能来自该池，池耗尽返回显式失败。
- 池分配与回收固定采用空闲链表头弹出/头插回收；`in_use` 只承担正式句柄合法性校验，不参与“找下一个可分配槽位”。
- 删除 `owns_storage` 及其相关分支，统一为“资源获取 / 运行态重置 / 资源释放”三段式生命周期。
- 把 `main.c`、调度器创建路径、适配层绑定函数和测试辅助全部改成通过正式生命周期入口申请句柄。
- 为正常路径与异常路径新增或重写 `contract` / `unit` / `integration` 测试，重点验证池耗尽、重复释放、无效句柄、重置与释放分离以及实例复用后的干净状态。

## 池容量模型

- 池容量是私有编译期常量，不进入公开 API，也不允许运行期扩容。
- 当前阶段必须先把“容量有限且耗尽会失败”的语义固定下来，再决定常量具体取值。
- 若现有主路径只需要单实例，容量可以先取最小可行值；若测试或受控集成路径需要并存多实例，则通过提高私有常量满足，但不改变公开接口和异常语义。
- 所有测试不得依赖具体数值，只能依赖“占满后继续申请会失败、释放后可再次申请成功”的正式行为。

## 嵌入式 C 安全计划

**是否适用**：是  
**命名与分层**：继续使用 `snake_case`、宏全大写、类型 `_t` 结尾；公开头文件只暴露声明与正式接口；真实结构体、池状态和容量常量只留在私有头或实现文件  
**格式与常量**：4 空格缩进、禁 Tab、控制语句带花括号；池容量和资源状态使用具名常量  
**错误策略**：获取失败、重复释放、非法句柄、无效重置和运行态误用都返回显式错误结果；沿用现有 `operation_result_t` / `ERROR_CODE_*` 语义体系  
**等待策略**：本特性不新增等待或阻塞路径；生命周期接口本身不引入新的超时模型  
**中断策略**：不适用；本次不新增 ISR 或硬件中断逻辑  
**并发与内存安全**：保持单进程单主控使用模型；通过正式占用状态和句柄合法性校验避免外部误持有；所有指针在装配与运行前执行空值校验  
**状态机**：生命周期状态至少覆盖“可获取 -> 已占用 -> 占用中可重置 -> 已释放”；异常路径不得污染其他槽位或其他已占用实例  
**注释风格**：公开接口补齐中文 Doxygen；内部关键池管理逻辑只在必要处添加简洁中文注释
