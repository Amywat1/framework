# 功能规格：洗车主控运行骨架

**功能分支**：`003-wash-controller-runtime`  
**创建日期**：2026-05-06  
**状态**：草稿  
**输入**：用户描述：“把当前代码从可测试执行内核做成可运行主控骨架，支持常驻主控进程、正式输入通道、持续推进主循环、启动前预检、状态查询和日志增强，并补齐关键集成/仿真测试。”

## 澄清、假设与权衡（必填）

- **假设**：同一时刻仍只允许一个运行中的洗车任务；现有洗车会话、执行状态、等待条件、超时和优先级规则继续有效；主控可以长期保留运行时上下文而不要求每单重建进程；正式输入通道只要求成为真实运行入口，不要求本次扩展到云端或多进程边界。
- **未决问题**：不适用
- **解释选择**：本规格将“常驻主控进程”解释为启动后持续等待输入和超时、在任务结束后保持可接单；将“正式输入通道”解释为脱离测试注入方式、面向真实运行的统一事件入口；将“状态查询”解释为外部使用方可以在任意时刻读取当前会话状态、执行状态、等待条件和最近原因。
- **不在范围内**：车辆识别、动态规划、云端/OTA、完整运行模式系统、完整故障中心、多进程拆分、性能优化专项。

## Clarifications

### Session 2026-05-06

- Q: 本阶段“正式输入通道”的主形态是什么？ → A: 主控进程通过行文本标准输入接收 `start / stop / feedback / fault / status` 命令。
- Q: 主控对 `stdin` 命令的正式输出方式是什么？ → A: 每条 `stdin` 命令都在 `stdout` 返回一条单行文本结果，`status` 返回当前状态单行摘要。
- Q: 遇到未知命令或格式错误命令时，主控应如何处理？ → A: 返回单行错误结果并记录原因，主控继续运行并等待下一条命令。
- Q: 当没有活动任务时收到 `stop / feedback / fault`，主控应如何响应？ → A: `fault` 记为全局故障，`stop / feedback` 忽略。
- Q: 记录了“全局故障状态”后，后续 `start` 应该怎么处理？ → A: 有全局故障时拒绝 `start`，直到通过后续明确动作清除故障。
- Q: `fault` 的“显式清除形式”具体采用哪种命令格式？ → A: 使用 `fault clear` 作为清除全局故障的唯一正式命令格式。
- Q: 当没有活动任务但存在全局故障时，`status` 单行摘要必须至少包含什么？ → A: 必须包含无活动任务、全局故障存在和最近全局故障原因。
- Q: `start` 命令的正式格式应是什么？ → A: 使用 `start <program_id>`，启动时必须显式指定程序。
- Q: `feedback` 命令的正式格式应是什么？ → A: 使用 `feedback <signal_code>`。
- Q: 用于“上报故障”的 `fault` 命令正式格式应是什么？ → A: 使用 `fault <fault_code> <reason>`。

## 用户场景与测试（必填）

### 用户故事 1 - 常驻主控连续接单（优先级：P1）

作为洗车设备的运行方，我希望主控启动后持续驻留并在一单结束后继续等待下一单，这样系统才能成为真正可运行的主控骨架，而不是只执行一次就退出的测试程序。

**为什么是这个优先级**：如果主控不能常驻运行并连续接单，后续输入、超时推进、状态查询和联调都没有实际承载体。

**独立验证方式**：启动主控后连续提交多次洗车任务，确认第一单结束后进程不退出，第二单和后续任务可在同一次运行周期内继续创建和完成。

**验收场景**：

1. **给定** 主控已经启动且当前没有运行中的任务，**当** 主控进入空闲等待，**则** 进程保持运行并持续保留可查询的当前上下文。
2. **给定** 第一单洗车任务已经完成或中止，**当** 后续新的启动输入到达，**则** 主控在不重启进程的情况下创建新的任务并进入下一轮处理。
3. **给定** 主控已经连续处理过多单任务，**当** 当前没有新输入，**则** 主控继续等待而不是自行退出。

---

### 用户故事 2 - 通过正式输入与预检驱动任务（优先级：P1）

作为洗车设备的操作方，我希望通过主控进程的行文本标准输入通道提交 `start <program_id>`、`stop`、`feedback <signal_code>`、`fault <fault_code> <reason>` 和状态查询命令，并通过标准输出获得每条命令对应的单行处理结果；其中 `fault` 命令既能上报故障，也能通过 `fault clear` 显式清除全局故障，同时在启动前完成必要预检，这样任务推进边界完整、输入来源真实且启动风险可控。

**为什么是这个优先级**：正式输入和启动前预检决定主控是否具备真实可运行价值，也是把当前执行内核接入长期运行骨架的最小闭环。

**独立验证方式**：通过主控进程的行文本标准输入通道提交 `start <program_id>`、`stop`、`feedback <signal_code>`、`fault <fault_code> <reason>`、`fault clear` 和 `status` 命令，并检查标准输出返回的单行结果；其中验证 `fault <fault_code> <reason>` 的故障上报与 `fault clear` 的显式清除两种形式；同时构造预检通过与失败场景，确认系统只在预检通过后启动任务，其他输入都通过同一入口进入明确处理路径。

**验收场景**：

1. **给定** 主控空闲且启动前预检通过，**当** 收到启动输入，**则** 系统创建新任务并进入运行中。
2. **给定** 主控空闲但启动前预检失败，**当** 收到启动输入，**则** 系统拒绝创建任务、记录失败原因，并保持可继续等待下一次启动。
3. **给定** 当前任务处于运行中并等待外部条件，**当** 收到停止、匹配反馈或故障输入，**则** 系统按既定规则推进当前任务或结束当前任务。

### 用户故事 3 - 持续可观测并覆盖关键边界（优先级：P2）

作为联调和维护人员，我希望主控在运行中可以持续看到任务状态、执行状态、等待条件和最近原因，并且关键场景有自动化覆盖，这样才能快速判断问题并降低主控骨架上线前的边界风险。

**为什么是这个优先级**：主控骨架的核心风险在状态迁移边界，不先建立观察与回归能力，后续联调很难定位问题。

**独立验证方式**：在正常、停止、故障、超时和连续任务场景中执行状态查询并检查关键迁移记录，确认状态可读、原因可追踪、自动化测试能覆盖主要路径。

**验收场景**：

1. **给定** 当前任务正在运行或等待，**当** 发起状态查询，**则** 能直接获得当前会话状态、执行状态、等待条件和最近原因。
2. **给定** 当前任务发生关键迁移（如启动、停止、故障、超时、完成、拒绝启动），**当** 查询最近记录，**则** 能看到对应结果和原因。
3. **给定** 主控骨架的自动化验证执行，**当** 覆盖正常、停止、故障、超时和连续任务场景，**则** 每个场景都有明确可核对的预期结果。

### 边界场景

- 当主控处于空闲等待且长时间没有输入时，系统必须保持运行并继续等待，而不是因“无事可做”退出。
- 当上一单已经结束但迟到反馈、重复反馈或陈旧故障输入随后到达时，系统不得重新打开已结束任务，也不得污染下一单的状态。
- 当当前已有运行中任务时再次收到新的启动输入，系统必须拒绝该启动并记录原因。
- 当收到 `start` 命令但未提供 `program_id` 时，系统必须拒绝该命令并返回单行错误结果，不得创建任务。
- 当收到 `feedback` 命令但未提供 `signal_code` 时，系统必须拒绝该命令并返回单行错误结果，不得推进任务状态。
- 当收到 `fault` 上报命令但未提供 `fault_code` 或 `reason` 时，系统必须拒绝该命令并返回单行错误结果，不得伪造故障状态迁移。
- 当启动前预检失败时，系统必须拒绝本次启动，但主控进程必须继续保持可用并等待后续新的启动。
- 当当前任务处于等待状态且没有新事件到达时，系统必须继续由等待超时驱动后续推进，不得卡死在等待中。
- 当状态查询发生在“无任务”“运行中”“已结束但尚未接到下一单”三种时刻时，系统都必须返回可判断的当前状态快照。
- 当主控收到未知命令或格式错误命令时，系统必须返回单行错误结果、记录原因，并继续保持运行，不得因输入错误退出。
- 当没有活动任务时收到 `fault` 命令时，系统必须将其记录为全局故障；当没有活动任务时收到 `stop` 或 `feedback` 命令时，系统必须忽略，不得创建任务或伪造任务级状态迁移。
- 当主控存在未清除的全局故障状态时，系统必须拒绝新的 `start` 命令，直到通过后续明确动作清除故障；在此之前不得创建新任务。
- 当主控存在全局故障状态且收到 `fault clear` 命令时，系统必须清除该全局故障并恢复后续 `start` 的可受理性。
- 当没有活动任务但存在全局故障状态时，`status` 单行摘要必须明确体现“无活动任务”“全局故障存在”以及最近全局故障原因。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须以常驻主控的形式运行，启动后在没有显式外部终止的情况下不得因为单次任务结束或暂时无输入而退出。
- **FR-002**：系统必须在一次主控运行周期内持续保留运行时上下文，并在任务切换时只重置任务级状态，不丢失继续接收下一单所需的主控能力。
- **FR-003**：系统必须支持主控进程通过行文本标准输入通道接收至少五类命令：`start <program_id>`、`stop`、`feedback <signal_code>`、`fault <fault_code> <reason>` 和 `status`；其中 `fault clear` 必须作为清除全局故障的唯一正式命令格式。
- **FR-003A**：`start` 命令必须显式包含 `program_id`；当未提供 `program_id` 时，系统必须拒绝该命令并返回单行错误结果。
- **FR-003B**：`feedback` 命令必须显式包含 `signal_code`；当未提供 `signal_code` 时，系统必须拒绝该命令并返回单行错误结果。
- **FR-003C**：用于上报故障的 `fault` 命令必须显式包含 `fault_code` 和 `reason`；当任一参数缺失时，系统必须拒绝该命令并返回单行错误结果。
- **FR-004**：系统必须将行文本标准输入通道接收到的业务命令统一转换为主控可处理的标准触发或查询请求，并进入同一套推进规则。
- **FR-004A**：系统必须对每条通过行文本标准输入收到的命令，在标准输出返回一条可判断的单行文本结果；其中 `status` 命令必须返回当前状态单行摘要。
- **FR-004C**：当没有活动任务但存在全局故障状态时，`status` 命令返回的单行摘要必须至少包含“无活动任务”“全局故障存在”以及最近全局故障原因。
- **FR-004B**：当收到未知命令或格式错误命令时，系统必须在标准输出返回单行错误结果并记录原因，同时保持主控继续运行并等待下一条命令。
- **FR-005**：系统必须持续推进主循环，反复处理输入事件、等待超时以及它们之间的优先级竞争，直到当前任务进入明确结果。
- **FR-006**：当当前任务结束后，系统必须自动回到可接收下一单的等待状态，而不是要求重启主控或重建全部上下文。
- **FR-007**：系统必须在每次启动任务前执行预检，至少覆盖本次启动所需的资源条件和传感器条件。
- **FR-008**：当启动前预检失败时，系统必须拒绝创建新任务、记录失败原因，并保持主控继续可用。
- **FR-008A**：当主控存在未清除的全局故障状态时，系统必须拒绝新的 `start` 命令，不得创建新任务，直到通过后续明确动作清除该全局故障。
- **FR-009**：当正式输入通道收到停止、反馈或故障输入时，系统必须在当前状态允许的前提下进入对应处理路径；不匹配、迟到或重复输入不得错误推进流程。
- **FR-009A**：当没有活动任务时收到 `fault` 命令，系统必须将其记录为全局故障；当没有活动任务时收到 `stop` 或 `feedback` 命令，系统不得创建任务或推进任务状态。
- **FR-009B**：当主控存在全局故障状态且收到 `fault clear` 命令时，系统必须清除该全局故障并记录清除结果。
- **FR-010**：当当前任务处于等待状态且没有新输入到达时，系统必须继续依赖等待超时推进后续路径，直到任务完成、降级结束或中止。
- **FR-011**：系统必须继续遵守已定义的触发优先级竞争规则，保证同一处理窗口只收敛到一个明确结果。
- **FR-012**：系统必须提供状态查询能力，使外部使用方能够读取当前是否存在活动任务、当前会话状态、当前执行状态、当前等待条件和最近原因。
- **FR-013**：系统必须记录关键迁移和关键拒绝结果，至少覆盖启动接受、启动拒绝、预检失败、停止处理、故障处理、等待超时、任务完成和任务中止。
- **FR-014**：系统必须在连续任务场景下正确隔离不同任务的状态、原因和结果，避免上一单残留信息直接污染下一单。
- **FR-015**：系统必须提供自动化验证，至少覆盖正常完成、人工停止、故障处理、等待超时、连续任务和预检失败场景。
- **FR-016**：系统必须保证主控骨架的正式输入通道成为这些场景的主要驱动入口，不得主要依赖测试代码直接向执行内核注入事件来证明主流程成立。

### 范围与简化要求（必填）

- **SS-001**：本次只把当前执行内核接成可长期运行的主控骨架，覆盖常驻运行、正式输入、持续推进、启动前预检、状态查询与日志增强。
- **SS-002**：本次不得扩展到车辆识别、动态规划、云端/OTA、完整运行模式系统、完整故障中心、多进程拆分或专项性能优化。
- **SS-003**：本次必须尽量复用现有会话、执行、等待、优先级和日志模型，仅在支撑长期运行主控骨架所必需的范围内扩展职责。
- **SS-004**：本次引入的全局故障状态只承担主控骨架级最小职责：记录空闲 `fault`、阻断后续 `start`、支持 `fault clear` 清除；不得在本期扩展为通用故障分类、分级处置、联动恢复或完整故障中心能力。

### 嵌入式 C 与安全要求（适用时填写）

- **EC-001**：C 代码命名必须遵循文件名/函数名/变量名使用 `snake_case`、宏全大写、类型以 `_t` 结尾、全局变量加 `g_`、静态变量加 `s_`
- **EC-002**：头文件必须只放声明，不得定义变量，并提供防重复包含保护
- **EC-003**：代码必须使用 4 空格缩进、禁 Tab、控制语句必须带花括号、用具名常量替代魔法数字
- **EC-004**：变量必须初始化；关键返回值、指针判空、共享数据保护和缓冲区边界必须检查
- **EC-005**：硬件/通信/存储/升级相关操作必须定义明确错误返回，以及重试、降级、告警或上报、复位或停止策略
- **EC-006**：所有等待都必须定义超时和最大重试次数
- **EC-007**：中断处理必须保持轻量，禁止阻塞、复杂计算、日志和动态内存
- **EC-008**：关键控制逻辑必须定义显式状态机、上电默认状态、故障安全状态、日志和关键参数校验
- **EC-009**：代码注释必须使用中文；所有对外公开接口必须有中文 Doxygen 注释，并覆盖用途、参数、返回值和必要约束；硬件相关接口与回调接口同样适用

### 关键实体（若功能涉及数据则必填）

- **主控运行实例**：表示一次主控进程生命周期内持续存在的运行载体，包含长期保留的运行时上下文、当前可接单状态以及最近一次处理结果。
- **正式输入事件**：表示通过真实输入通道进入主控的启动、停止、反馈和故障输入，包含事件类型、事件内容、到达时点和处理结果；故障上报输入至少包含 `fault_code` 和 `reason`。
- **命令响应结果**：表示主控对单条输入命令返回的单行文本响应，包含命令处理结论以及与当前状态相关的必要摘要信息。
- **全局故障状态**：表示不依附于当前任务的主控级故障记录，用于表达“当前无活动任务时收到 fault”这类全局异常状态及其最近原因；本期仅用于记录、阻断后续 `start` 和支持 `fault clear` 清除。
- **启动预检结果**：表示某次启动前对资源和传感器检查后的判断结果，包含是否允许启动以及失败原因。
- **状态快照**：表示某一时刻可对外查询的主控视图，包含当前任务存在性、会话状态、执行状态、等待条件、最近原因，以及适用时的全局故障存在性和最近全局故障原因。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：在自动化验证中，主控启动后能够在同一次运行周期内连续处理至少 3 次洗车任务，无需重启且不会因单次任务结束而退出。
- **SC-002**：在覆盖启动、停止、反馈和故障四类正式输入的验证中，100% 的输入都通过正式输入通道进入系统，并产生可核对的接受、拒绝或忽略结果。
- **SC-003**：在预检失败场景验证中，100% 的启动请求都在任务创建前被拒绝，并能在一次状态查询或最近记录中直接看到失败原因。
- **SC-003A**：在“空闲时收到 `fault` 后再次收到 `start`”的验证中，100% 的启动请求都被拒绝，直到全局故障被明确清除。
- **SC-004**：在正常完成、停止、故障、超时和连续任务场景中，100% 的任务都收敛到单一明确结果，且主控在每单结束后都能回到等待下一单的状态。
- **SC-005**：在运行中和空闲时的观测验证中，维护人员都能在 30 秒内通过单次状态查询和最近记录判断当前会话状态、执行状态、等待条件和最近原因。

## 验证方式（必填）

- **V-001**：启动主控后连续执行多次洗车任务，验证主控常驻、第一单结束后不退出且下一单可继续创建和完成。
- **V-002**：通过正式输入通道分别提交启动、停止、反馈和故障，验证输入都进入统一处理路径并产生可核对结果。
- **V-003**：构造启动前预检通过与失败两类场景，验证通过时可启动任务，失败时拒绝启动并记录原因。
- **V-003A**：在无活动任务时注入 `fault`，随后提交 `start`，验证系统拒绝启动并保持主控运行，直到全局故障被明确清除。
- **V-003B**：在主控存在全局故障状态时提交 `fault clear`，验证系统清除全局故障并允许后续新的 `start` 被受理。
- **V-004**：在等待阶段不注入反馈，验证系统可持续依赖超时推进到预定义结果，而不是停留在无界等待。
- **V-005**：在正常、停止、故障、超时和连续任务场景中执行状态查询与最近记录检查，验证状态、等待条件和最近原因都可直接读取。

## 依赖与默认前提

- 现有洗车会话、执行状态、等待条件、优先级竞争和日志记录能力可继续作为主控骨架的基础能力复用。
- 主控所在环境能够持续向系统提供启动、停止、反馈和故障这四类输入来源。
- 启动前预检所依赖的资源状态和传感器状态在当前系统边界内可被读取并形成可判断结果。
- 当前业务边界仍是单机单车位和单活任务，本次不引入排队、多并发洗车或远程协调能力。
