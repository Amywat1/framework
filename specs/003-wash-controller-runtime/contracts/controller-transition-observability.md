# 接口契约：主控关键迁移与可观测性

## 1. 说明

本契约定义主控骨架在运行中必须可记录和可查询的关键迁移范围，供日志、状态查询和测试共同遵循。

## 2. 必记录事件

| 事件类别 | 最小记录内容 |
|----------|--------------|
| 启动受理 | `program_id`、结果、时间 |
| 启动拒绝 | 拒绝原因、时间 |
| 启动前预检失败 | 失败原因、时间 |
| 运行中停止 | 触发源、结果、时间 |
| 运行中故障 | 故障代码、原因、结果、时间 |
| 等待超时 | 等待对象、超时结果、时间 |
| 任务完成 | 终态结果、时间 |
| 任务中止 | 中止原因、时间 |
| 空闲 fault | 全局故障代码、原因、时间 |
| fault clear | 清除结果、时间 |
| 错误命令 | 原始命令、错误原因、时间 |

## 3. 最近原因语义

- `latest_reason_code` 必须总能指向最近一次关键迁移、关键拒绝或关键错误。
- `global_fault_reason` 独立于任务级最近原因保存。
- 当新任务开始时，任务级最近原因允许被新任务上下文覆盖，但全局故障原因必须按清除规则独立管理。

## 4. 查询与日志一致性

- `status` 返回的最近原因必须与最近关键记录一致。
- 测试验证应能通过状态查询或日志观察到相同的关键结果。
- 对被忽略命令的处理也应具备可追踪结果，至少能反映在最近原因或日志中。

## 5. 契约约束

- 同一任务终态只能收敛为一个明确结果。
- 关键迁移记录不得把超时、启动失败或全局故障错误记成无关触发类型。
- 连续任务之间的最近原因和终态记录必须隔离，不能直接污染下一单。
