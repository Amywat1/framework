# 接口契约：主控 `stdin/stdout` 命令协议

## 1. 说明

本契约定义洗车主控运行骨架对外暴露的正式命令接口。输入来源固定为 `stdin` 行文本，输出来源固定为 `stdout` 单行文本。

## 2. 通用规则

- 一行输入代表一条命令。
- 每条命令必须返回且只返回一条单行输出。
- 未知命令、格式错误命令和被拒绝命令都必须返回可判断的单行结果。
- 命令处理失败不得导致主控退出。

## 3. 输入命令格式

| 命令 | 格式 | 说明 |
|------|------|------|
| 启动 | `start <program_id>` | 创建新的洗车任务 |
| 停止 | `stop` | 请求停止当前任务 |
| 反馈 | `feedback <signal_code>` | 提交设备或业务反馈 |
| 故障上报 | `fault <fault_code> <reason>` | 上报故障 |
| 故障清除 | `fault clear` | 清除全局故障 |
| 状态查询 | `status` | 查询当前状态摘要 |

## 4. 参数约束

| 命令 | 约束 |
|------|------|
| `start` | 必须提供 `program_id` |
| `feedback` | 必须提供 `signal_code` |
| `fault <fault_code> <reason>` | 必须同时提供 `fault_code` 和 `reason` |
| `fault clear` | 只能写成两段形式，且不得带其他参数 |
| `stop` / `status` | 不允许带业务参数 |

## 5. 单行输出格式

建议输出结构如下：

```text
result=<result_code> accepted=<true|false> detail=<summary>
```

最小约束：

- 必须包含结果结论
- 必须是单行
- 必须足以区分“受理 / 拒绝 / 忽略 / 错误 / 状态摘要”

## 6. 典型结果语义

| 场景 | 结果要求 |
|------|----------|
| `start` 成功 | 返回已受理，并附带程序或会话摘要 |
| `start` 因预检失败被拒绝 | 返回拒绝，并带失败原因 |
| `start` 因全局故障被拒绝 | 返回拒绝，并带全局故障原因 |
| 空闲时 `stop` / `feedback` | 返回忽略 |
| 空闲时 `fault` | 返回已记录全局故障 |
| `fault clear` 成功 | 返回已清除 |
| 未知命令 | 返回错误 |
| `status` | 返回状态摘要 |

## 7. 契约约束

- `status` 不得推进业务状态。
- 全局故障存在时，新的 `start` 必须返回拒绝。
- `fault clear` 是清除全局故障的唯一正式命令。
- 同一命令不得产生多条输出。
