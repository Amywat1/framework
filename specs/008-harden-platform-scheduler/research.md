# Phase 0 研究：平台层调度器收敛

## 决策 1：引入平台层 `controller_scheduler` 抽象，主程序只启动调度器

**Decision**  
新增平台层 `controller_scheduler` 抽象接口，对上层只暴露配置、启动、停止、可观测输出和事件源启停语义；`main.c` 只负责创建 `system_context_t`、装配端口、构造调度配置并启动调度器。

**Rationale**  
当前 `main.c` 同时承担组合根、Linux 等待循环和退出处理，导致平台细节外泄。将等待/唤醒/退出壳层收敛到调度器后，可以让上层只依赖“启动控制器运行时”这一语义，而不感知 `select`、文件描述符和超时驱动细节。

**Alternatives considered**

- 保留 `main.c` 外层循环，仅抽一个辅助函数：仍会让主程序感知平台事件分发细节。
- 直接用 `main_loop_run()` 代表调度器：无法表达事件源启停、退出策略和观测对象，也会混淆单拍推进与运行时壳层职责。

## 决策 2：Linux 运行时使用 `epoll + timerfd + eventfd + CLOCK_MONOTONIC`

**Decision**  
Linux 平台实现采用 `epoll` 统一等待，使用 `timerfd` 产生固定控制周期，使用 `eventfd` 接收外部通知或线程唤醒，所有时间基准统一使用 `CLOCK_MONOTONIC`。

**Rationale**  
这组原语天然匹配“周期主时钟 + 外部唤醒 + 单线程消费”的模型。`timerfd` 能把控制周期显式化，避免继续使用 `wait_timeout` 混充线程唤醒时钟；`eventfd` 便于把采集线程或异步通知收敛成统一唤醒源；`CLOCK_MONOTONIC` 能避免系统时钟跳变污染控制节拍。

**Alternatives considered**

- 继续使用 `select` + `stdin` + 动态超时：无法表达统一事件源注册，也让周期语义继续依赖业务等待条件。
- 使用 `poll`：在本场景下不能比 `epoll` 带来更清晰的事件源注册模型。
- 使用信号或 `signalfd` 作为主唤醒机制：不适合作为控制周期的唯一主时钟，也会引入额外信号处理约束。

## 决策 3：领域推进始终单线程，`pthread` 仅可用于采集/通知

**Decision**  
领域状态推进保持单线程，只允许调度线程调用 `main_loop_advance_time()` 与 `main_loop_run()`。如确有采集或外设回调需要脱离调度线程，则 `pthread` 仅负责刷新最新快照缓存并通过 `eventfd` 发出通知。

**Rationale**  
用户明确要求“单线程领域推进模型”。允许采集线程存在，但限制其职责为“缓存更新 + 唤醒”，可以保留硬件接入灵活性，同时避免多条异步路径同时推进会话/执行状态机。

**Alternatives considered**

- 多线程并发推进领域逻辑：会立即破坏现有单拍状态机模型与测试假设。
- 完全禁止任何辅助线程：在需要外部采集回调或阻塞 I/O 时会把所有等待压力重新压回调度线程，不利于扩展。

## 决策 4：固定控制周期是稳定态唯一主时钟，命令只触发受控补跑

**Decision**  
稳定态只由周期事件驱动推进时间并执行一次或有限次 `main_loop_run()`；高优先级命令事件入队后可以触发一次受控补跑；退出流程只允许有界排空。

**Rationale**  
当前 `drain_runtime()` 的“只要还有事就一直跑”不适合周期化控制器。新的策略要求每拍工作量有上界，并把“实时稳定”优先级放在“尽快排空所有积压”之前。

**Alternatives considered**

- 沿用无限 `drain_runtime()`：会导致命令堆积时单拍失控，难以保证周期稳定性。
- 命令到来后完全不补跑：会让某些人机交互反馈过慢，不利于 CLI/管理口体验。

## 决策 5：外部事件源按命令/通知/退出三类建模

**Decision**  
所有平台外部输入统一分为命令事件源、通知事件源、退出事件源。命令事件负责解析为正式命令或触发请求，通知事件只更新缓存/唤醒，退出事件负责进入排空或直接退出。

**Rationale**  
这三类语义正好覆盖用户指定的扩展点，并能防止未来把各类输入逻辑继续堆回 `main.c` 或 CLI 适配层。明确分类后，也更容易为每类事件源建立计数、优先级和失败语义。

**Alternatives considered**

- 只区分“CLI”与“非 CLI”：不足以表达通知和退出的独立语义。
- 所有输入统一抽象成触发队列元素：会鼓励通知路径直接驱动领域状态，违背“通知不直接改领域状态”的原则。

## 决策 6：`process_formal_command` 不再拥有持续推进循环

**Decision**  
正式命令处理路径继续负责命令解析、入队请求和结果投影，但不再以 `while (has_pending_trigger_id)` 方式直接循环调用 `main_loop_run()`；是否补跑一拍由调度器决定。

**Rationale**  
若命令路径继续自带推进循环，就会绕过“平台调度器统一决定何时推进”的新边界。将补跑时机收回调度器后，命令事件才能与周期、通知、退出一起纳入同一套优先级和有界工作量策略。

**Alternatives considered**

- 保持现状：命令入口仍然是事实上的第二调度器。
- 完全拆掉正式命令用例，只在平台层直接改状态：会破坏现有应用层边界，得不偿失。

## 决策 7：调度器观测复用现有日志体系并新增节拍指标

**Decision**  
调度器观测通过现有日志/状态输出体系扩展，至少记录周期次数、超拍次数、连续超拍次数、各类事件源触发次数、挂起触发数量和最近错误原因。

**Rationale**  
本次是运行时架构变更，最容易出问题的是节拍抖动、积压和退出路径。复用现有日志体系可以减少引入新依赖，同时让运维信息保持在同一诊断通道中。

**Alternatives considered**

- 只在调试输出中打印：信息不可结构化，也不利于契约验证。
- 新增独立指标系统：对当前项目过重，超出本期范围。

## 决策 8：测试重点从业务结果扩展到调度语义

**Decision**  
沿用现有 CTest 分层，新增/调整 contract、integration、simulation 测试，覆盖周期驱动、命令补跑、通知只更新缓存、退出排空、多事件优先级和观测输出。

**Rationale**  
这次最大的风险不是功能“跑不起来”，而是 ownership 和时序变了但表面结果仍能通过少量业务测试。必须用契约验证把运行时边界守住。

**Alternatives considered**

- 仅补单元测试：无法覆盖多事件顺序、退出策略和主程序组合根收敛。
- 只靠手工运行验证：无法长期防止架构回退。
