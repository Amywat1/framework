# 契约：文件仓储程序加载路径

## 目标
确保 `file_program_repository` 在标准 JSON 库迁移后仍然只通过同一条程序导入路径加载配置文件，不出现并行旧路径或行为漂移。

## 路径契约
- 仓储继续使用 `<config_root>/programs/<program_id>.json` 组装配置路径。
- 当运行时缓存程序存在且 `program_id` 命中缓存时，继续优先返回缓存对象。
- 当缓存未命中时，仓储必须调用迁移后的 `json_program_parser_parse(...)`。

## 成功契约
- 解析成功时，`load_program_impl(...)` 返回成功状态，并填充输出 `wash_program_t`。
- 通过仓储加载的程序对象与直接调用 `json_program_parser_parse(...)` 的结果保持等价。
- 通过 `test_load_runtime_program_from_fixture(...)` 注入运行时缓存后，再次经仓储读取应得到等价对象。

## 失败契约
- 当文件不存在、JSON 语法非法、重复键、字段错误、旧 `stages` 顶层结构或语义非法时，仓储加载必须返回失败。
- 仓储失败时不得伪造默认程序，也不得污染运行时缓存。

## 兼容要求
- 本次迁移不改变仓储的公开端口形状。
- `file_program_repository_set_runtime_program(...)` 的运行时注入能力保持不变。
- 测试辅助与运行时加载路径继续共享同一套程序导入实现。

## 验证关注点
- 直接仓储加载合法配置成功。
- 直接解析、仓储加载和测试辅助注入三条路径得到等价程序对象。
- 非法夹具经测试辅助导入失败后，不会在仓储侧留下可读取的运行时缓存。
