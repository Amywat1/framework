# 功能规格：标准化 JSON 配置导入

**功能分支**：`011-json-parser-migration`  
**创建日期**：2026-05-11  
**状态**：草稿  
**输入**：用户描述：`"当前程序对json文件的信息导入是通过自定义函数解析的，需要调整成调用标准的json解析库，原来的自定义解析函数删除"`

## 澄清、假设与权衡（必填）

- **假设**：现有洗车程序在 `segments` 契约下的 JSON 文件结构、字段语义和领域校验规则在本次变更中保持不变；本次只替换 JSON 语法解析方式，不重定义 `segments` 配置契约。
- **假设**：现有通过 JSON 导入成功的 `segments` 程序文件、测试夹具和仓储读取路径仍然是本次迁移后的兼容基线；旧 `stages` 语义不再保留，属于应删除的遗留输入。
- **假设**：调用方仍通过现有程序导入入口加载配置；本次不新增第二套导入入口，也不要求调用方改写业务流程。
- **未决问题**：不适用。
- **解释选择**：将“使用标准 JSON 解析库”解释为“停止维护项目自带的通用 JSON 语法解析实现，改为依赖成熟、标准化的 JSON 解析能力”，因为用户诉求的重点是降低自定义解析维护成本并提升一致性，而不是重写领域映射规则。
- **不在范围内**：不修改 `segments` 程序契约下的业务字段定义，不新增配置能力，不重构主控 runtime 生命周期，不调整调度器、领域状态机或事件日志语义。

## 用户场景与测试（必填）

### 用户故事 1 - 稳定导入有效程序配置（优先级：P1）

作为程序配置的维护者，我希望系统继续稳定导入现有合法的 `segments` JSON 程序文件，但底层改用标准化解析能力，这样我可以在不改动当前正式程序定义的前提下继续使用现有配置。

**为什么是这个优先级**：如果有效配置在迁移后不能继续导入，主控的程序加载能力会立即回退，迁移就不具备可交付价值。

**独立验证方式**：使用当前已知合法的程序配置文件执行导入，确认导入成功且得到的程序标识、版本和段配置与迁移前一致。

**验收场景**：
1. **给定** 一个当前已被系统接受的合法程序 JSON 文件，**当** 执行程序导入，**则** 系统应成功生成与迁移前等价的程序对象。
2. **给定** 仓储按程序标识读取合法 JSON 文件，**当** 主控请求加载该程序，**则** 系统应成功返回可供后续运行使用的程序数据。

---

### 用户故事 2 - 显式拒绝无效配置（优先级：P1）

作为程序配置的维护者，我希望语法错误、重复字段、缺失必需字段或违反当前契约的 JSON 文件继续被明确拒绝，这样无效配置不会被静默接受或形成半有效程序数据。

**为什么是这个优先级**：迁移解析方式后如果错误配置被错误放行，会直接破坏当前配置契约和运行安全边界。

**独立验证方式**：分别导入语法非法、重复键、缺失关键字段和语义不合法的配置文件，确认系统返回明确失败，且不会产出可继续使用的程序对象。

**验收场景**：
1. **给定** 一个 JSON 语法不合法或字段结构不符合当前契约的程序文件，**当** 执行程序导入，**则** 系统应返回明确失败结果。
2. **给定** 一个包含重复关键字段或违反当前程序约束的程序文件，**当** 执行程序导入，**则** 系统应拒绝导入且不保留部分导入结果。

---

### 用户故事 3 - 停止维护自定义 JSON 解析器（优先级：P2）

作为代码维护者，我希望程序配置导入路径不再依赖项目自带的通用 JSON 解析实现，这样后续维护只需要关注程序配置到领域模型的映射与校验，而不是继续维护底层 JSON 语法解析细节。

**为什么是这个优先级**：这决定了本次迁移是否真正收敛维护面；如果只是包一层外观而保留旧解析器，目标并未达成。

**独立验证方式**：检查构建、仓储调用和测试入口，确认程序导入路径已统一依赖标准化解析能力，旧的自定义通用 JSON 解析实现及其编译引用已被移除。

**验收场景**：
1. **给定** 程序配置导入相关的生产代码、测试代码和构建清单，**当** 审查导入依赖路径，**则** 不应再发现项目自带的通用 JSON 语法解析实现被继续编译或调用。

### 边界场景

- 当程序 JSON 文件不存在、无法读取或内容为空时，系统必须返回明确失败，而不是构造默认程序对象。
- 当 JSON 文件包含当前契约不接受的重复字段时，系统必须继续拒绝导入，避免标准解析能力默许覆盖行为而改变既有约束。
- 当 JSON 语法解析成功但输入仍使用旧 `stages` 语义、或存在业务字段缺失、类型错误、顺序约束不满足时，系统必须继续按当前 `segments` 配置契约返回失败。
- 当导入失败时，系统不得留下可被后续仓储读取或运行流程误用的部分程序数据。
- 当测试夹具覆盖合法边界值时，迁移后的导入结果必须与迁移前保持一致，不得因解析方式变化而改变既有可接受输入范围。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须将程序配置文件的 JSON 语法解析切换为标准 JSON 解析库提供的能力，而不是继续使用项目自定义的通用 JSON 语法解析实现。
- **FR-002**：系统必须保留当前程序配置导入入口的职责边界，使调用方仍通过同一条程序导入路径完成文件读取、JSON 解析、领域映射和结果返回。
- **FR-003**：系统必须保证当前已接受的合法 `segments` 程序 JSON 配置在迁移后仍可成功导入，并生成与迁移前等价的程序领域数据。
- **FR-004**：系统必须保证当前已拒绝的无效配置在迁移后继续被拒绝，包括但不限于语法错误、重复字段、缺失必需字段、类型不匹配和违反程序顺序或控制规则的情况。
- **FR-005**：系统必须在任一导入失败场景下返回明确失败结果，不得输出部分有效的程序对象，也不得将失败延后到运行期暴露。
- **FR-006**：系统必须继续以当前 `segments` 程序配置契约为准执行字段提取、枚举解释、可选字段默认处理和业务校验，不得借本次迁移改变 `segments` JSON 结构或字段语义。
- **FR-006A**：系统必须将旧 `stages` 语义视为已废弃输入；迁移后不得继续把 `stages` 解释为合法程序结构，也不得把它纳入兼容范围。
- **FR-007**：系统必须让仓储加载路径、测试辅助入口和相关验证路径统一复用迁移后的程序导入实现，不得保留并行的旧解析路径。
- **FR-008**：系统必须从代码库、构建清单和调用路径中移除旧的自定义通用 JSON 解析实现，避免新旧两套语法解析逻辑长期并存。
- **FR-009**：系统必须在导入失败时保持程序仓储和运行上下文处于未加载有效程序的安全状态，避免后续流程误以为导入已成功。

### 范围与简化要求（必填）

- **SS-001**：系统必须只实现程序配置 JSON 导入迁移所需的最小改动，包括解析入口替换、旧解析器移除以及必要的回归验证。
- **SS-002**：系统不得在本轮中扩展 `segments` 程序配置格式、增加新字段、改变现有业务规则，或顺带恢复/兼容旧 `stages` 语义，亦不得顺带重构与 JSON 导入无关的主控模块。
- **SS-003**：系统必须将改动限制在程序配置导入模块、依赖该导入的仓储/测试调用点、构建清单以及验证当前导入契约的相关测试范围内。

### 关键实体（若功能涉及数据则必填）

- **程序配置文档**：存放在文件系统中的 JSON 程序定义，是程序导入的原始输入，包含程序标识、版本和分段控制信息。
- **导入后的程序对象**：由配置文档转换得到的强类型程序数据，供仓储和主控运行流程后续使用。
- **导入失败结果**：表示配置读取、JSON 语法解析、字段提取或业务校验任一环节失败的明确结果，用于阻止无效程序进入运行路径。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：100% 的现有合法 `segments` 程序配置夹具在迁移后仍能成功导入，并保持与迁移前等价的程序核心数据。
- **SC-002**：100% 的现有无效程序配置夹具在迁移后仍能被明确拒绝，不出现静默放行或崩溃。
- **SC-003**：程序导入相关的生产入口、仓储入口和测试入口全部收敛到同一套迁移后导入路径，不再存在旧自定义通用 JSON 解析路径被继续调用。
- **SC-004**：代码库中不再保留供程序导入路径使用的项目自定义通用 JSON 语法解析实现，维护重点收敛到配置映射与业务校验。

## 验证方式（必填）

- **V-001**：使用当前合法 `segments` 程序配置和合法边界值夹具执行导入，验证导入成功以及关键程序数据保持一致。
- **V-002**：使用语法错误、重复字段、缺失字段、类型错误和语义非法夹具执行导入，验证每类场景都返回明确失败。
- **V-003**：验证仓储读取和测试辅助入口调用的都是迁移后的统一导入实现，而不是保留旧解析路径。
- **V-004**：检查构建清单与源文件引用，验证旧自定义通用 JSON 解析实现已被删除且不再参与编译。

## 依赖与默认前提

- 项目在当前构建环境中可以接入一个标准 JSON 解析库，并满足现有程序配置导入所需的基本对象、数组、字符串、数字和布尔值解析能力。
- 现有测试夹具和配置样例可以作为迁移前后行为等价性的基线。
- 程序导入后的领域映射与校验逻辑仍由当前代码库维护，本次只替换其底层 JSON 语法解析来源。
