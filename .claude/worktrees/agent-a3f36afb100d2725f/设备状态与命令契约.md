# 设备状态与命令契约

## 1. 范围

- 本文只整理当前仓库已实现、且已由现有代码与测试共同固定下来的对外契约。
- 若与历史草案冲突，以当前正式入口和状态查询实现为准：
  - `src/application/use_cases/process_formal_command.c`
  - `src/application/use_cases/process_wash_trigger.c`
  - `src/application/use_cases/query_wash_session_status.c`
- 本文不描述未来规划，不重写 `domain-design.md`。

当前正式命令如下：

- `status`
- `homing`
- `start <program_id>`
- `stop`
- `fault <code> <reason>`
- `fault clear`

当前统一响应外壳如下：

```text
result=<result_code> accepted=<true|false> detail=<detail>
```

`status` 也是单行响应，但其 `detail` 会展开为状态摘要。

## 2. 当前设备状态定义

| 状态 | 对外显示值 | 含义 | 关键约束 |
| --- | --- | --- | --- |
| `INIT` | `init` | 主控上下文已创建但尚未完成正式初始化。 | 当前正式 runtime 完成初始化后会立即进入 `STOPPED`，联调通常不会长期停留在此态；若在此态收到 `start`，会以 `device_state_init` 拒绝。 |
| `STOPPED` | `stopped` | 设备已停止，但尚未完成开机后或异常后的重新归位。 | 这是正式联调入口常见初始态；`start` 不允许，必须先执行 `homing`。 |
| `RECOVERING` | `recovering` | 正在执行 `homing` 流程。 | 当前实现中是同步过渡态，不作为稳态停留；成功进入 `IDLE`，失败进入 `EXCEPTION`。 |
| `IDLE` | `idle` | 设备已归位，允许受理新任务。 | `start` 只允许在该态进入；`homing` 在该态会被拒绝。 |
| `RUNNING` | `running` | 当前存在运行中的洗车会话。 | `stop` 和 `fault <code> <reason>` 可受理；`start`、`homing`、`fault clear` 均拒绝。 |
| `EXCEPTION` | `exception` | 当前被异常态阻塞。 | 不保证一定伴随 `global_fault_present=true`；离开该态的正式命令是 `fault clear`，成功后只回到 `STOPPED`，不会自动回到 `IDLE`。 |

补充约束：

- `device_state` 与 `session_state` 独立存在。
- `fault clear` 或 `homing` 不会自动把既有 `session_state=aborted/completed` 清空。
- 当前实现里，`STOPPED` 表示“已停机但未就绪”，`IDLE` 才表示“可启动新任务”。

## 3. 命令语义

| 命令 | 当前语义 | 成功后的主结果 |
| --- | --- | --- |
| `status` | 只读查询，不推进业务状态。 | 返回状态摘要；不刷新 `latest_result_code` / `latest_reason_code`。 |
| `homing` | 执行恢复/归位流程：先 `stop_all`，再执行 `home_roof_brush` 并校验反馈。 | `device_state: STOPPED -> RECOVERING -> IDLE`，`latest_result_code=accepted`，`latest_reason_code=homing_completed`。 |
| `start <program_id>` | 启动新任务。 | `device_state: IDLE -> RUNNING`，`latest_result_code=accepted`，`latest_reason_code=session_started`。 |
| `stop` | 人工停止当前运行任务。 | `device_state: RUNNING -> STOPPED`，`session_state -> aborted`，`latest_result_code=aborted`，`latest_reason_code=manual-stop`。 |
| `fault <code> <reason>` | 上报故障。 | 非运行中：记录全局故障并进入 `EXCEPTION`；运行中：中止当前任务并进入 `EXCEPTION`。 |
| `fault clear` | 清除异常阻塞并把设备退回可恢复起点。 | `device_state: EXCEPTION -> STOPPED`，`latest_result_code=accepted`，`latest_reason_code=global_fault_cleared`。 |

`start <program_id>` 的额外前提：

- `device_state` 必须已经是 `IDLE`
- 不存在运行中的会话
- 不存在活跃全局故障
- `program_id` 必须存在且配置有效

## 4. 命令受理矩阵

说明：

- `受理` 表示命令会进入当前正式处理路径。
- `拒绝(<reason_code>)` 表示当前实现会返回该稳定拒绝原因。
- `fault <code> <reason>` 在所有设备态都可受理，但后续落点取决于是否存在运行中会话。

| 设备状态 | `status` | `homing` | `start <program_id>` | `stop` | `fault <code> <reason>` | `fault clear` |
| --- | --- | --- | --- | --- | --- | --- |
| `INIT` | 受理 | 拒绝(`homing_requires_stopped`) | 拒绝(`device_state_init`) | 拒绝(`stop_requires_running`) | 受理，进入 `EXCEPTION` | 拒绝(`fault_clear_requires_exception`) |
| `STOPPED` | 受理 | 受理 | 拒绝(`device_state_stopped`) | 拒绝(`stop_requires_running`) | 受理，进入 `EXCEPTION` | 拒绝(`fault_clear_requires_exception`) |
| `RECOVERING` | 受理 | 拒绝(`homing_requires_stopped`) | 拒绝(`device_state_recovering`) | 拒绝(`stop_requires_running`) | 受理，进入 `EXCEPTION` | 拒绝(`fault_clear_requires_exception`) |
| `IDLE` | 受理 | 拒绝(`homing_requires_stopped`) | 有条件受理 | 拒绝(`stop_requires_running`) | 受理，进入 `EXCEPTION` | 拒绝(`fault_clear_requires_exception`) |
| `RUNNING` | 受理 | 拒绝(`homing_requires_stopped`) | 拒绝(`device_state_running`) | 受理 | 受理，中止会话并进入 `EXCEPTION` | 拒绝(`fault_clear_requires_exception`) |
| `EXCEPTION` | 受理 | 拒绝(`homing_requires_stopped`) | 拒绝(`device_state_exception`) | 拒绝(`stop_requires_running`) | 受理，保持或刷新 `EXCEPTION` | 受理，进入 `STOPPED` |

`IDLE` 下 `start <program_id>` 的补充拒绝条件：

- `running_session_exists`
- `global_fault_active`
- `program_unavailable`
- `program_invalid`

`fault clear` 的补充约束：

- 当前实现只检查 `device_state == EXCEPTION`。
- 不额外检查 `global_fault_present` 是否为 `true`。
- 成功后不会自动执行 `homing`，也不会自动回到 `IDLE`。

## 5. 当前稳定的关键 `reason_code`

下表只收录当前外部联调最关键、且已有代码路径或测试覆盖固定下来的值。

### 5.1 协议与参数层

| 场景 | `reason_code` |
| --- | --- |
| 空命令 | `empty_command` |
| 不支持的命令 | `unsupported_command` |
| `start` 缺少 `program_id` | `start_requires_program_id` |
| `homing` 带多余参数 | `homing_takes_no_argument` |
| `stop` 带多余参数 | `stop_takes_no_argument` |
| `status` 带多余参数 | `status_takes_no_argument` |
| `fault` 缺少故障码 | `fault_requires_code` |
| `fault <code>` 缺少原因 | `fault_requires_reason` |
| `fault clear` 带多余参数 | `fault_clear_takes_no_argument` |

### 5.2 `start <program_id>`

| 场景 | `reason_code` |
| --- | --- |
| 在 `INIT` 启动 | `device_state_init` |
| 在 `STOPPED` 启动 | `device_state_stopped` |
| 在 `RECOVERING` 启动 | `device_state_recovering` |
| 在 `RUNNING` 启动 | `device_state_running` |
| 在 `EXCEPTION` 启动 | `device_state_exception` |
| 设备不在空闲就绪态的统一语义 | `device_not_idle` |
| 已存在运行中会话 | `running_session_exists` |
| 全局故障仍活跃 | `global_fault_active` |
| 程序不存在 | `program_unavailable` |
| 程序无效 | `program_invalid` |
| 启动成功 | `session_started` |

说明：对外正式命令在非 `IDLE` 时会优先返回 `device_state_*` 系列拒绝码；`device_not_idle` 是触发处理层仍保留的稳定语义。

### 5.3 `homing`

| 场景 | `reason_code` |
| --- | --- |
| 非 `STOPPED` 发起 | `homing_requires_stopped` |
| 成功完成 | `homing_completed` |
| `stop_all` 失败 | `stop_all_failed` |
| 屋刷回零命令失败 | `roof_home_command_failed` |
| 运行态快照不可读 | `runtime_snapshot_unavailable` |
| 回零反馈超时 | `roof_home_feedback_timeout` |
| 回零反馈存在但未到位 | `roof_home_not_reached` |
| 恢复过程中触发全局故障 | `recovering_alarm_triggered` |
| 恢复失败兜底码 | `homing_failed` |

### 5.4 `stop`

| 场景 | `reason_code` |
| --- | --- |
| 非 `RUNNING` 发起 | `stop_requires_running` |
| 正式 `stop` 命令成功 | `manual-stop` |

说明：测试辅助会直接提交自定义停止原因，但对外 `stop` 命令当前固定使用 `manual-stop`。

### 5.5 `fault <code> <reason>` / `fault clear`

| 场景 | `reason_code` |
| --- | --- |
| 非运行中记录故障成功 | `global_fault_recorded` |
| 运行中故障中止 | `<fault_code>` 原样进入 `latest_reason_code` |
| 非 `EXCEPTION` 清故障 | `fault_clear_requires_exception` |
| 清故障成功 | `global_fault_cleared` |

说明：

- 非运行中提交 `fault <code> <reason>` 时，`global_fault_reason` 保存的是 `<reason>`，而最近原因码固定为 `global_fault_recorded`。
- 运行中提交 `fault <code> <reason>` 时，最近原因码固定为 `<fault_code>` 本身，例如 `fault-e-stop`。

### 5.6 运行期完成/异常补充

| 场景 | `reason_code` |
| --- | --- |
| 程序正常完成 | `program_finished` |
| 段超时导致异常结束 | `segment_timeout` |

## 6. 三条核心业务流转链路

### 6.1 正常完成

```text
STOPPED
  -- homing --> RECOVERING --> IDLE
  -- start <program_id> --> RUNNING
  -- 程序完成 --> IDLE
```

关键观测点：

- `homing` 成功后：`latest_result_code=accepted`，`latest_reason_code=homing_completed`
- `start` 成功后：`device_state=running`，`session_state=running`，`latest_reason_code=session_started`
- 程序完成后：`device_state=idle`，`session_state=completed`，`latest_result_code=completed`，`latest_reason_code=program_finished`

### 6.2 人工停止

```text
STOPPED
  -- homing --> RECOVERING --> IDLE
  -- start <program_id> --> RUNNING
  -- stop --> STOPPED
```

关键观测点：

- `stop` 成功后：`device_state=stopped`
- `session_state=aborted`
- `latest_result_code=aborted`
- `latest_reason_code=manual-stop`

### 6.3 故障恢复

```text
RUNNING
  -- fault <code> <reason> --> EXCEPTION
  -- fault clear --> STOPPED
  -- homing --> RECOVERING --> IDLE
```

关键观测点：

- 故障进入 `EXCEPTION` 后：`global_fault_present=true`
- 若故障发生在运行中：`session_state=aborted`，`latest_result_code=aborted`，`latest_reason_code=<fault_code>`
- `fault clear` 后：`device_state=stopped`，`latest_reason_code=global_fault_cleared`
- `homing` 后：`device_state=idle`，但既有 `session_state=aborted` 不会因此自动清空

## 7. 对外可观测字段

### 7.1 `status` 当前直接可见字段

当前 `status` 单行摘要格式如下：

```text
result=status accepted=true detail=device=<device_state> session=<session_id|none> state=<session_state> execution=<execution_state> lifecycle=<lifecycle_state> stage=<stage_id|none> wait=<wait_reason|none> global_fault=<true|false> reason=<latest_reason_code|none>
```

可直接联调的关键字段：

| 字段 | 当前来源 | 含义 |
| --- | --- | --- |
| `device_state` | `status detail.device=` | 当前设备状态，值为 `init/stopped/recovering/idle/running/exception` |
| `session_state` | `status detail.state=` | 当前会话状态，值为 `none/created/running/completed/aborted` |
| `latest_reason_code` | `status detail.reason=` | 最近一次关键迁移、拒绝或错误原因 |
| `global_fault_present` | `status detail.global_fault=` | 当前是否存在活跃全局故障 |

### 7.2 当前已稳定但不直接展开在 `status detail` 里的字段

| 字段 | 当前来源 | 含义 |
| --- | --- | --- |
| `latest_result_code` | 正式命令响应外壳中的 `result=`；以及运行时最近结果投影 | 最近一次对外处理结果，如 `accepted`、`rejected`、`aborted`、`completed` |
| `global_fault_reason` | `query_wash_session_status` 状态视图 | 最近一次全局故障的人类可读原因文本 |
| `session_id` | `status detail.session=` | 当前会话标识；无活动会话时为 `none` |
| `stage_id` | `status detail.stage=` | 当前执行段标识 |
| `wait_reason` | `status detail.wait=` | 当前等待原因；无等待时为 `none` |

补充说明：

- `status` 查询本身是只读操作，不应改变 `latest_result_code`、`latest_reason_code`、队列深度或业务状态。
- 当前实现里，`latest_result_code` 的主对外载体是正式命令响应外壳，不是 `status detail`。

## 8. 当前联调最容易混淆的三点

- `STOPPED` 不是“可启动”，而是“已停机但未就绪”；要到 `IDLE` 才能 `start`。
- `fault clear` 只负责把 `EXCEPTION` 拉回 `STOPPED`；要重新开机链路，仍需再做一次 `homing`。
- 设备恢复到 `IDLE` 不代表会话结果被清空；异常后的 `session_state=aborted` 会保留到下一次新会话覆盖或系统重置。
