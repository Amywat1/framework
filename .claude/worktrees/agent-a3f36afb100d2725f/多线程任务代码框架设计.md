# 多线程任务代码框架设计

## 1. 文档目的

本文档用于固化当前讨论过的多线程任务框架方案，重点回答以下问题：

- 在引入多线程后，如何保持业务逻辑不依赖具体操作系统。
- 平台层、业务层、装配层分别负责什么。
- 线程、`job`、队列、事件流、`runtime` 之间的关系应该如何划分。
- 报警检测、主控制流程、命令输入、周期推进应如何协作。

本文档是目标设计说明，不要求与当前仓库已有实现完全一致。后续若按本方案演进，应遵循“最小改动、逐步落地”的原则。

## 2. 设计前提

### 2.1 业务前提

- 设备后续会同时存在主控制流程和报警检测流程。
- 报警判断属于业务逻辑，不应写入平台层。
- 设备主状态机仍应集中控制，不能由多个线程同时修改。

### 2.2 技术前提

- 平台层可以使用线程、任务、阻塞队列、互斥锁、条件变量等并发机制。
- 业务层不直接依赖 `pthread`、RTOS task 或其他具体系统 API。
- 当前目标是先建立清晰框架，不预支复杂扩展点。

### 2.3 约束结论

- 平台层只提供通用执行能力和硬件访问能力。
- 业务层提供各类 `job` 的实际处理逻辑。
- `runtime` 只能由一个控制任务写入。

## 3. 总体思路

推荐采用“三层分离”：

1. 平台层：提供通用 `worker_thread`、队列、时间、IO 读取、执行器输出。
2. 业务层：实现 `io_read_job`、`alarm_detect_job`、`control_job` 等任务逻辑。
3. 装配层：创建对象并将业务 `job` 注入平台线程壳子中运行。

核心原则不是“线程越多越好”，而是：

- 线程负责并发执行。
- `job` 负责业务处理。
- `control_job` 负责集中修改业务状态。

## 4. 分层边界

### 4.1 平台层职责

平台层只回答“怎么跑起来”，不回答“业务上应该怎么处理”。

平台层负责：

- 创建、停止、等待线程或任务。
- 提供阻塞队列、互斥锁、条件变量、信号量等并发基础设施。
- 提供时间接口，例如当前时间、周期等待、睡眠。
- 读取硬件原始 IO 快照。
- 接收外部命令输入。
- 输出执行器底层动作。

平台层不应出现以下业务概念：

- `device_state`
- `homing`
- `running`
- `exception`
- `alarm_evaluator`
- `runtime_coordinator`

### 4.2 业务层职责

业务层只回答“事件来了以后该怎么处理”。

业务层负责：

- 根据原始 IO 和状态视图判断是否形成报警。
- 消费命令事件、报警事件、周期事件并推进状态机。
- 管理报警锁存、清除、重复抑制等规则。
- 决定设备状态切换与执行器动作时机。

业务层不应直接：

- 调用具体线程创建 API。
- 依赖某个操作系统的同步原语头文件。
- 读写平台线程对象内部状态。

### 4.3 装配层职责

装配层只负责接线，不负责业务判断。

装配层负责：

- 创建 `runtime`、队列、视图存储、各类 `job` 上下文。
- 将业务 `job` 封装为平台线程可执行入口。
- 启动和停止所有线程。

## 5. 依赖方向

正确的依赖方向如下：

```text
业务层 -> 平台抽象接口
平台实现层 -> 平台抽象接口
装配层 -> 业务层 + 平台实现层
```

禁止出现以下依赖：

```text
平台实现层 -> 业务实现层
```

这意味着：

- 平台线程壳子不能直接包含业务状态机头文件。
- 平台层不能直接调用 `alarm_evaluator`、`runtime_coordinator` 等业务函数。
- 业务逻辑应通过 `job` 入口注入到平台线程中运行。

## 6. 线程与 job 的关系

### 6.1 基本原则

- 线程是平台提供的执行壳子。
- `job` 是业务层定义的任务逻辑。
- 平台层只知道“运行某个入口函数”，不知道该入口函数代表什么业务。

### 6.2 推荐关系

当前场景建议优先使用“一线程一 job”的最小版本，便于理解和调试：

- `io_read_thread` 运行 `io_read_job`
- `alarm_detect_thread` 运行 `alarm_detect_job`
- `command_read_thread` 运行 `command_read_job`
- `tick_thread` 运行 `tick_job`
- `control_thread` 运行 `control_job`

其中：

- 线程生命周期由平台层管理。
- `job` 的输入输出、阻塞策略和业务处理由业务层定义。

### 6.3 为什么平台层只保留通用 worker_thread

如果平台层直接实现“报警检测线程”，并在其中调用业务判断函数，就会形成“平台依赖业务”的反向耦合。

更合理的做法是：

- 平台层只提供通用 `worker_thread`
- 业务层提供 `alarm_detect_job`
- 装配层把 `alarm_detect_job` 注入 `worker_thread`

## 7. 队列与事件流

### 7.1 推荐队列

建议至少定义以下队列：

| 队列 | 用途 | 生产者 | 消费者 |
| --- | --- | --- | --- |
| `io_snapshot_queue` | 传递原始 IO 快照 | `io_read_job` | `alarm_detect_job` |
| `alarm_event_queue` | 传递报警事件 | `alarm_detect_job` | `control_job` |
| `control_event_queue` | 传递命令事件与周期事件 | `command_read_job`、`tick_job` | `control_job` |

### 7.2 推荐事件分类

控制输入至少分为三类：

- `command_event`
  - `HOMING`
  - `START`
  - `STOP`
- `tick_event`
- `alarm_event`

其中：

- `alarm_event` 是业务报警判断后的结果，不是原始 IO。
- 原始 IO 只在 `io_snapshot_queue` 内传递，不直接进入主状态机。

### 7.3 推荐主链路

```text
io_read_thread
    -> io_snapshot_queue
    -> alarm_detect_job
    -> alarm_event_queue
    -> control_job
    -> runtime_coordinator
    -> runtime / actuator_port
```

```text
command_read_thread
    -> control_event_queue
    -> control_job
    -> runtime_coordinator
```

```text
tick_thread
    -> control_event_queue
    -> control_job
    -> runtime_coordinator
```

如果报警判断需要参考当前业务状态，再补一条只读链路：

```text
control_job -> runtime_view_store -> alarm_detect_job
```

### 7.4 推荐优先级

`control_job` 处理事件时，建议固定优先级如下：

1. `alarm_event`
2. `command_event`
3. `tick_event`

原因如下：

- 报警需要优先中断普通流程。
- 命令优先级应高于普通周期推进。
- `tick` 只负责推进既有流程，不应抢占安全相关处理。

## 8. runtime 单写原则

这是整个框架最重要的约束。

### 8.1 原则定义

- `runtime` 只能由 `control_job` 写入。
- 其他任何线程或 `job` 都不能直接修改 `runtime`。

### 8.2 直接收益

- 避免多线程状态竞争。
- 状态切换路径集中，便于验证和审查。
- 故障、停止、归位、运行等规则可收敛在一处维护。

### 8.3 配套只读视图

如果其他 `job` 需要参考当前业务状态，应只读取只读视图，而不是读取真实 `runtime`。

推荐最小只读视图为：

| 字段 | 用途 |
| --- | --- |
| `device_state` | 判断当前设备主状态 |
| `wash_active` | 判断洗车流程是否在进行 |
| `fault_latched` | 判断当前是否存在锁存故障 |

该视图通过 `runtime_view_store` 提供线程安全读取。

## 9. 模块职责

### 9.1 `io_read_job`

职责：

- 周期读取平台原始 IO 快照。
- 将快照写入 `io_snapshot_queue`。

不负责：

- 报警判断。
- 设备状态切换。

### 9.2 `alarm_detect_job`

职责：

- 从 `io_snapshot_queue` 读取原始输入。
- 读取 `runtime_view`。
- 调用报警判定逻辑生成 `alarm_event`。
- 将 `alarm_event` 写入 `alarm_event_queue`。

不负责：

- 直接停机。
- 直接修改 `runtime`。

### 9.3 `command_read_job`

职责：

- 阻塞读取外部命令。
- 将命令转换为 `command_event`。
- 写入 `control_event_queue`。

不负责：

- 判断当前状态是否允许执行该命令。

### 9.4 `tick_job`

职责：

- 周期产生 `tick_event`。
- 写入 `control_event_queue`。

不负责：

- 推进业务状态机。

### 9.5 `control_job`

职责：

- 消费报警、命令、周期事件。
- 作为 `runtime` 的唯一写者推进状态机。
- 调用业务协调器决定动作与状态流转。
- 刷新 `runtime_view_store`。

### 9.6 `alarm_evaluator`

职责：

- 根据原始 IO 和状态视图做业务报警判定。
- 处理持续时间、去抖、状态相关报警规则。
- 决定是否产生报警触发或报警清除事件。

### 9.7 `runtime_coordinator`

职责：

- 作为主状态机入口处理 `HOMING`、`START`、`STOP`、`ALARM`、`TICK`。
- 决定是否调用执行器端口。
- 决定如何修改 `runtime`。

### 9.8 `fault_manager`

建议作为后续补强模块，负责：

- 报警锁存
- 重复报警抑制
- 清除规则
- 故障信息更新

如果第一阶段规则较简单，可以先合并在 `runtime_coordinator` 内，后续再拆分。

## 10. 业务事件职责归口

在当前洗车机场景下，建议将关键事件统一收口到 `runtime_coordinator`：

| 事件 | 发现来源 | 规则判断者 | 修改 `runtime` 的位置 |
| --- | --- | --- | --- |
| `HOMING` | `command_read_job` | `runtime_coordinator` | `control_job` 内部调用 `runtime_coordinator` |
| `START` | `command_read_job` | `runtime_coordinator` | `control_job` 内部调用 `runtime_coordinator` |
| `STOP` | `command_read_job` | `runtime_coordinator` | `control_job` 内部调用 `runtime_coordinator` |
| `ALARM` | `alarm_detect_job` | `alarm_evaluator` + `runtime_coordinator` | `control_job` 内部调用 `runtime_coordinator` |
| `TICK` | `tick_job` | `runtime_coordinator` | `control_job` 内部调用 `runtime_coordinator` |

补充约束：

- `alarm_evaluator` 只负责判断“是否形成报警”。
- `runtime_coordinator` 负责判断“报警后状态怎么变、是否停机”。
- 任何 `job` 都不能绕过 `control_job` 直接修改主状态。

## 11. 推荐最小落地版本

当前最推荐的最小版本如下：

- `io_read_thread`
- `alarm_detect_thread`
- `control_thread`

配套最小对象如下：

- `io_snapshot_queue`
- `alarm_event_queue`
- `control_event_queue`
- `runtime`
- `runtime_view_store`
- `alarm_evaluator`
- `runtime_coordinator`

说明如下：

- `command_read_job` 和 `tick_job` 可以继续保留为独立线程，也可以视平台情况并入其他事件源。
- `fault_manager` 可以暂不单独拆分。
- 不建议一开始就引入复杂任务注册中心、通用 worker 池或大而全事件总线。

## 12. 推荐实现顺序

建议按下面顺序逐步落地，确保每一步都形成可验证闭环。

### 12.1 第一步：统一核心数据

- 定义 `runtime`
- 定义 `runtime_view`
- 定义 `command_event`、`tick_event`、`alarm_event`

验证目标：

- 状态与事件模型命名稳定。
- 主状态机输入输出边界明确。

### 12.2 第二步：补齐平台通用能力

- 定义 `worker_thread`
- 定义阻塞队列接口
- 定义时间接口
- 定义原始 IO 与执行器端口接口

验证目标：

- 平台接口中不出现业务状态名。

### 12.3 第三步：先打通控制主链路

- 实现 `control_job`
- 实现 `runtime_coordinator`
- 先接入 `command_event` 和 `tick_event`

验证目标：

- `HOMING`、`START`、`STOP`、`TICK` 可以在单控制线程内跑通。

### 12.4 第四步：接入原始 IO 与报警判定

- 实现 `io_read_job`
- 实现 `alarm_detect_job`
- 实现 `alarm_evaluator`
- 接入 `alarm_event_queue`

验证目标：

- 报警路径可以从原始 IO 一直推进到 `EXCEPTION`。

### 12.5 第五步：补强故障管理

- 视复杂度决定是否拆出 `fault_manager`
- 补齐报警锁存、清除、重复抑制等细节

验证目标：

- 故障处理语义稳定，便于扩展更多报警源。

## 13. 关键注意事项

### 13.1 不要把平台层写成业务层

以下做法不推荐：

- 在平台线程源码中直接调用业务状态机函数。
- 在平台层中引入 `device_state`、`alarm_evaluator` 这类业务概念。

### 13.2 不要破坏 runtime 单写原则

以下做法不推荐：

- `alarm_detect_job` 直接改 `runtime`
- IO 线程发现异常后直接切设备状态
- 多个线程同时持有 `runtime` 可写引用

### 13.3 不要过早引入复杂抽象

当前阶段不建议：

- 通用 worker 池
- 动态 `job` 注册中心
- 复杂优先级调度框架
- 全量 `runtime` 镜像同步

当前场景更适合先用最少线程、最少队列建立稳定闭环。

### 13.4 不要把原始输入和业务报警混为一层

应明确区分：

- 原始 IO 快照
- 报警业务判定结果
- 主状态机处理后的故障状态

这三层职责混在一起，会使后续排查和扩展都变困难。

### 13.5 命名应直指职责

建议优先使用以下命名风格：

- `io_read_job`
- `alarm_detect_job`
- `control_job`
- `runtime_view_store`
- `runtime_coordinator`

不建议使用语义过泛或过大的命名，例如：

- `manager`
- `engine`
- `processor`

除非职责边界已经非常明确。

## 14. 结论

当前讨论下，最稳妥且便于逐步落地的方案是：

- 平台层只提供通用线程壳子和并发基础设施。
- 业务层把每类任务实现成独立 `job`。
- `control_job` 作为 `runtime` 唯一写者集中推进状态机。
- 通过 `io_snapshot_queue`、`alarm_event_queue`、`control_event_queue` 串联多线程任务。
- 先用最小版本建立闭环，再按实际复杂度决定是否继续拆分。

这套方案的重点不在“线程越多越好”，而在“线程负责并发执行，业务状态仍然集中控制”。
