# 研究记录：洗车工步真实控制模型

## 决策 1：继续保持 `C17 + Linux + CMake + ctest` 朴素技术栈

**Decision**：继续使用 C17、Linux、CMake、ctest 和现有自定义测试程序体系，不引入新的语言运行时或更重的测试框架。  
**Rationale**：
- 当前仓库已经稳定运行在该栈上，和驱动、IO、位置采集、传感器接口集成成本最低。
- 单进程主控更容易保持控制周期、时序可见性和故障定位能力。
- 现有测试目录已经覆盖单元、契约、集成、仿真与 HIL smoke，升级测试语义即可。  
**Alternatives considered**：
- 引入更重的测试框架：收益不足，且会把工作从工艺模型转向工具迁移。
- 引入更高层运行时：会削弱嵌入式主控的可控性。

## 决策 2：采用“配置适配层 + 强类型领域模型 + 应用编排 + 平台运行时”四层分离

**Decision**：在现有 `adapters / domain / application / platform / main` 结构内，明确四层职责：`adapters/config` 只做配置映射，`domain/model + domain/services` 只做强类型业务规则，`application/use_cases + coordinators` 只做编排，`platform/*` 只做现场事实输入与控制输出。  
**Rationale**：
- 这与仓库现有分层一致，不需要目录级大搬迁。
- 最能守住“字符串不进领域、平台不解释工艺、应用不重新长成大杂烩”的边界。
- 允许在不改技术栈的情况下，把当前阶段模型直接升级为工步段模型。  
**Alternatives considered**：
- 新建更复杂的中台层或总线层：会制造额外抽象层，没有当前需求支撑。
- 继续在 `system_context` 内混合推进：会延续职责渗透和状态写入混乱。

## 决策 3：用强类型工步段模型直接替换 `wash_stage` 粗粒度模型

**Decision**：领域层不再以 `wash_stage_t` 作为真实工艺核心，而是改为 `wash_program_v1_t / wash_segment_t / continuous_control_t / conditional_control_t / exit_action_t / exception_policy_t` 等强类型结构表达工步段。  
**Rationale**：
- 当前 `wash_stage_t` 只能表达简单移动、刷组和 `rinse_enabled`，无法承载条件控制、退出动作和收尾失败语义。
- 工步段模型需要显式区分段目标、运行控制、完成条件和退出动作，继续在旧结构上加字段会让语义更乱。
- 规格已明确旧模型冲突时不要求兼容，因此应直接替换而不是双轨并存。  
**Alternatives considered**：
- 继续扩展 `wash_stage_t`：会让“阶段”名义上存在、实质上承载工步段复杂语义，命名和职责都失真。
- 设计通用动作脚本：调试与审查成本更高，不适合有限控制对象。

## 决策 4：JSON 只允许存在于配置适配层，运行期统一使用枚举与固定结构体

**Decision**：`json_program_parser` 重构为严格映射器，所有 `"forward" / "tail" / "follow" / "abort_session"` 等文本都在 `adapters/config` 一次性转为枚举、数值、布尔和固定结构体；领域层和应用层禁止继续解释 JSON 字符串。  
**Rationale**：
- 当前解析器仍依赖字符串探测和内建程序回退，不能表达真实工艺，也无法拒绝非法组合。
- 适配层是处理字段合法性、枚举映射和能力边界的唯一合适位置。
- 这样后续字段名调整或新增条件类型时，修改面只会停留在配置适配层。  
**Alternatives considered**：
- 应用层按需解释 JSON：会让配置字段名扩散进主控核心。
- 平台层直接消费 JSON：会把平台事实和工艺规则耦合到一起。

## 决策 5：执行主链路改为“进入段 -> 运行段 -> 退出段”的持续评估模型

**Decision**：应用层统一按工步段生命周期推进：进入段初始化段状态，运行段持续判断位置与条件控制，退出段执行正式收尾动作并等待完成；不再沿用“阶段开始时一次性下发动作然后等待反馈”的旧模式。  
**Rationale**：
- 真实工艺依赖位置变化、延迟喷药、跟随控制和回零收尾，这些都发生在运行期持续评估中。
- 顶刷回零、停刷、停药、RO 水关闭和风干关闭都必须进入正式收尾语义。
- 把退出动作失败纳入状态机，比继续依赖临时硬编码更稳定。  
**Alternatives considered**：
- 在旧执行链路上继续插补条件判断：会让条件控制和退出动作持续旁路化。
- 把退出动作当作普通下一段：会模糊“本段完成”和“收尾完成”的边界。

## 决策 6：位置语义统一为可归一化的运行时事实 + 领域判定

**Decision**：平台层负责提供位置相关事实，领域层统一使用强类型位置语义判定车头/车尾到达、相对距离达到、绝对位置达到、回零完成等条件；同一条件控制规则必须明确使用相对位置或绝对位置。  
**Rationale**：
- 位置是平台事实，但“什么时候开始喷、什么时候停、什么时候完成本段”属于业务规则。
- 把位置字符串或随意信号名直接用作工艺规则，会导致控制对象错位。
- 归一化位置快照能让条件控制、完成条件和异常处理共享同一判断基础。  
**Alternatives considered**：
- 让各段自由解释位置：会制造不可追溯的启停点。
- 只保留车头/车尾文本判断：无法支持延迟喷药和区间控制。

## 决策 7：异常处理与退出失败统一由应用层编排收口

**Decision**：`fault / stop / timeout / exit_failure / position_lost / follow_lost` 等异常都由应用层统一收口，领域层只给出异常决议与退出策略；段内超时与收尾超时显式分离。  
**Rationale**：
- 规格明确异常可靠性比正常流程更关键。
- 一旦引入 `exit_actions`，退出失败就必须成为正式状态机语义的一部分。
- 应用层最适合协调异常决议、状态投影、最后结果和平台停机动作。  
**Alternatives considered**：
- 让每个领域服务直接停机或写结果：会重新分散异常语义。
- 只处理正常结束：无法覆盖回零失败、药剂关闭失败等关键场景。

## 决策 8：若 RO 水段和风干段执行机构能力缺失，本期直接补齐

**Decision**：把 RO 水与风干执行机构能力视为当前目标流程的正式组成部分。若现有 `actuator_port_t`、仿真驱动或状态反馈接口缺失，直接扩展端口和驱动，不保留占位能力。  
**Rationale**：
- 规格已明确 RO 水段和风干段是目标工步段，且缺失能力必须补齐。
- 如果只在配置层定义工步却没有对应执行机构，整个模型仍然是假的。
- 平台层扩展执行机构品类，比在应用层伪造阶段完成更符合分层职责。  
**Alternatives considered**：
- 把 RO 水/风干段先做成占位段：无法满足真实控制模型目标。
- 继续复用 `rinse_enabled` 一类布尔位：不足以表达正式段运行和关闭反馈。

## 决策 9：测试工具不升级，但测试语义升级为工步段流程与收尾动作

**Decision**：继续使用当前 `ctest + 自定义测试程序` 体系，但新增/调整测试覆盖工步段模型、配置映射、位置驱动条件控制、退出动作和异常收尾。  
**Rationale**：
- 现有测试基础已经成型，迁移工具没有必要。
- 本期真正变化的是业务语义，不是测试基础设施。
- 单元、集成、仿真和契约层都可以分别承载新模型验证。  
**Alternatives considered**：
- 优先升级测试框架：与核心业务改造目标无关。
- 只保留旧阶段测试：无法证明工步段模型真实可用。
