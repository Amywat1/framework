## 澄清、假设与权责（必填）

- **假设**：本次工作只面向当前单进程、单主循环控制器；现有 `select + timeout + stdin` 驱动方式、trigger 队列模型和正式命令对外语义都继续有效。
- **未决问题**：不适用
- **解释选择**：本规格将“DDD 架构基线加固”解释为修正依赖方向、收敛应用编排入口、明确核心状态所有权，并统一运行时结果表达；不把目标解释为目录大搬迁、抽象层扩张或业务语义重定义。
- **不在范围内**：不修改命令协议文本、配置格式、主循环基本策略、驱动端口设计和部署方式；不引入线程模型、消息总线、IPC 机制或新的正式入口路径；不要求一次性清除所有 `system_context` 使用点。

## 用户场景与测试（必填）

### 用户故事 1 - 领域层只表达事实与决策（优先级：P1）

作为后续扩展领域规则的开发者，我希望领域层只返回领域事实、决策结果和原因信息，而不是直接承担运行时记录或 CLI 表达职责，这样我在修改规则时不会被应用编排和外部表现细节牵连。

**为什么是这个优先级**：如果领域层继续反向依赖应用层记录职责，任何领域调整都会扩大改动面，破坏分层稳定性。

**独立验证方式**：检查领域服务对外返回的信息是否只包含领域结果、状态变化事实和决策原因；确认领域核心不再直接依赖应用层记录职责或 CLI 表达职责。

**验收场景**：
1. **给定** 某个触发事件需要领域服务做出接受、拒绝或等待决策，**当** 应用层调用领域服务时，**则** 领域层只返回领域结果与原因信息，不直接写入最近结果、日志或 CLI 响应。
2. **给定** 某个领域规则发生变更，**当** 评审其依赖关系时，**则** 可以确认领域层不需要穿透外部编排容器才能完成决策。

---

### 用户故事 2 - 应用层统一承接主触发编排（优先级：P1）

作为维护运行时主链路的开发者，我希望触发事件进入系统后先由应用层统一承接，再由应用层组织领域调用、结果记录和外部状态更新，这样主触发处理入口可以稳定为一个总路由与编排入口，而不是继续膨胀成细节混杂的大型实现。

**为什么是这个优先级**：主触发入口如果继续同时承担路由、细节决策、状态改写和外部表达，会让后续任何变更都集中撞在同一个高风险点上。

**独立验证方式**：沿典型触发事件检查主链路，确认主入口负责路由和高层编排，细节决策由下游职责承接。

**验收场景**：
1. **给定** 触发事件进入正式处理链路，**当** 系统开始受理该事件，**则** 存在一个稳定的应用层总入口负责统一路由与编排。
2. **给定** 需要处理接受、拒绝、忽略、状态迁移或超时决议，**当** 评审主入口职责时，**则** 看不到分散的细节写入和重复语义拼装。
3. **给定** `status` 命令从正式入口进入系统，**当** 系统处理该命令时，**则** 该命令由统一应用层入口承接，但保持只读查询语义且不进入 trigger 队列。

---

### 用户故事 3 - 核心运行时状态拥有清晰主写入边界（优先级：P1）

作为排查状态问题的维护者，我希望会话、执行、等待条件、全局故障和最近结果这些核心状态都能直接定位到主写入边界，这样我可以快速判断某个状态为什么被改动、应该由谁负责改动。

**为什么是这个优先级**：状态所有权不清晰会直接放大故障定位成本，也会让后续改动更容易引入越界写入。

**独立验证方式**：代码评审时逐一检查五类核心状态，确认每类状态都有明确主写入边界，并且领域层不再随意穿透整块运行时上下文改写无关状态。

**验收场景**：
1. **给定** 维护者检查会话、执行、等待条件、全局故障和最近结果的写入来源，**当** 逐类追踪时，**则** 每类状态都能定位到明确的主写入边界。
2. **给定** 某个领域服务只需要少量输入数据和端口，**当** 应用层向其发起调用时，**则** 不需要把整块 `system_context` 继续透传进领域核心。
3. **给定** 系统需要设置或清除 `global_fault`，**当** 处理相关事件时，**则** 只有统一主触发编排入口可以执行该写入。
4. **给定** 系统需要刷新最近结果，**当** 某个典型事件完成处理时，**则** 只有统一主触发编排入口可以写入最近结果。
5. **给定** 系统需要推进会话、执行、等待条件或处理超时，**当** 维护者追踪写入边界时，**则** 可以明确区分会话状态机、执行服务和超时服务各自的职责。

---

### 用户故事 4 - 对外运行时语义保持统一且稳定（优先级：P2）

作为联调和回归验证人员，我希望接受、拒绝、忽略、状态迁移、超时决议和故障清除等典型事件在最近结果、状态摘要、日志和 CLI 响应中的表达保持一致，这样我在排查问题时不需要对照多套互相冲突的语义。

**为什么是这个优先级**：即使结构被收敛，只要结果表达分散，对外行为仍然会表现为不稳定和难以解释。

**独立验证方式**：覆盖典型事件的回归路径，逐项比对最近结果、状态摘要、日志和 CLI 响应的表达是否一致，并确认正式命令外部行为未变。

**验收场景**：
1. **给定** 某个典型事件被系统处理，**当** 查看最近结果、状态摘要、日志和 CLI 响应，**则** 四者对该事件的结论与原因表达一致。
2. **给定** 现有正式命令路径执行回归，**当** 比对外部行为，**则** 当前用户可见语义保持稳定。
3. **给定** `status` 命令被正式入口受理，**当** 系统返回当前状态视图时，**则** 不改写最近结果，只呈现当前状态事实。

### 边界场景

- 当领域服务需要返回迁移结果、执行结果、等待结果或原因码时，系统必须允许这些领域事实返回到应用层，但不得让领域层直接承担外部记录职责。
- 当主触发入口处理复杂事件时，系统必须把细节决策、状态写入和语义映射下沉到明确职责边界，而不是继续堆积在总入口中。
- 当领域逻辑只需要少量运行时数据时，系统必须只传递必要数据和端口对象，而不是把 `system_context` 作为万能容器继续透传。
- 当某个典型事件既影响领域结论又影响对外表达时，系统必须在应用层统一完成最近结果、状态摘要、日志和 CLI 响应之间的映射。
- 当回归验证现有正式命令路径时，系统必须维持现有单进程、单主循环、`select + timeout + stdin` 驱动和 trigger 队列模型不变。
- 当 `status` 命令进入正式处理路径时，系统必须将其纳入统一应用层入口，但不得为了入口统一而改变其只读查询语义或强制其进入 trigger 队列。
- 当 `status` 命令被受理时，系统必须把它当作只读状态查询处理，不得将其记作最近一次业务处理结论。
- 当系统需要设置或清除 `global_fault` 时，系统必须通过统一主触发编排入口完成，而不是由其他服务分散直接改写。
- 当系统需要刷新最近结果时，系统必须通过统一主触发编排入口完成，而不是由领域服务或适配层顺带写入。
- 当系统推进会话、执行、等待条件和超时相关逻辑时，系统必须保持会话状态机、执行服务和超时服务之间的写入边界清晰，避免职责重叠。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须确保领域层只负责表达领域事实、决策结果、原因码和状态变化事实，不直接承担应用层记录职责。
- **FR-002**：系统必须确保领域层不再直接依赖应用层记录器、CLI 表达逻辑或其他外部表现形式。
- **FR-003**：系统必须提供一个稳定的应用层总入口来统一承接触发事件，并由该入口组织领域调用、结果记录和外部状态更新。
- **FR-003A**：系统必须将 `status` 命令纳入同一正式应用层入口，但保持其只读查询语义，并且不要求其进入 trigger 队列。
- **FR-004**：系统必须将当前主触发处理入口收敛为统一路由和高层编排入口，不再让其持续膨胀为同时承担细节决策、状态写入、日志表达和异常分支细节的实现。
- **FR-005**：系统必须为会话、执行、等待条件、全局故障和最近结果五类核心运行时状态定义清晰的主写入边界。
- **FR-006**：系统必须确保领域层不会随意穿透整块运行时上下文去改写无关状态，跨对象编排与最终结果落点由应用层负责。
- **FR-006A**：系统必须确保 `global_fault` 的设置与清除只由统一主触发编排入口负责，其他服务只能返回事实、判断或建议，不能直接改写该状态。
- **FR-006B**：系统必须确保最近结果只由统一主触发编排入口写入，领域服务和适配层不能直接写入最近结果。
- **FR-006C**：系统必须确保会话状态只由会话状态机写入，执行状态与等待条件只由执行服务写入，超时服务只产出超时决议而不直接接管其他运行时状态写入。
- **FR-007**：系统必须继续保留 `system_context` 在系统外壳和编排层中的使用能力，但领域服务只接收完成职责所必需的数据和端口对象。
- **FR-008**：系统必须让领域结果到最近结果、状态摘要、日志和 CLI 响应之间的映射集中在应用层统一处理。
- **FR-009**：系统必须对接受、拒绝、忽略、状态迁移、超时决议和故障清除等典型事件提供一致的对外结果表达。
- **FR-009A**：系统必须确保 `status` 只返回当前状态视图，不改写最近结果，也不把查询行为记录为新的业务处理结论。
- **FR-010**：系统必须保持当前单进程、单主循环、`select + timeout + stdin` 驱动方式和 trigger 队列模型不变。
- **FR-011**：系统不得引入新的线程模型、消息总线、IPC 机制或新的正式入口路径来完成本次结构调整。
- **FR-012**：系统必须将本次改动主要限定在应用层用例与协调器、领域服务以及少量入口适配代码上，而不以目录级大搬迁作为达成目标的前提。
- **FR-013**：系统必须在完成调整后仍保持现有正式命令路径和典型运行流程的外部行为稳定。

- **SS-001**：本次只收敛依赖方向、应用编排入口、核心状态所有权和运行时语义映射，不扩展新的业务能力。
- **SS-002**：本次不要求一次性清除所有 `system_context` 使用点，只要求其逐步退出领域核心。
- **SS-003**：本次不要求做目录级大搬迁，也不为追求形式纯度引入多余抽象。

- **EC-001**：新增或调整的公开接口必须继续遵守项目现有 C 分层和命名约束，并保持头文件只声明、实现留在源文件中的边界。
- **EC-002**：与状态推进和超时决议相关的逻辑必须继续维持显式边界检查和错误处理，不得因编排收敛而弱化现有安全约束。
- **EC-003**：公开接口和关键职责边界的注释必须继续满足项目既有中文 Doxygen 约束，便于后续评审与排障。

### 关键实体（若功能涉及数据则必填）

- **领域决策事实**：表示领域层返回给应用层的迁移结果、执行结果、等待结果、原因码和状态变化事实。
- **应用编排入口**：表示统一承接触发事件并组织领域调用、结果记录和外部状态更新的高层入口。
- **运行时状态所有权**：表示会话、执行、等待条件、全局故障和最近结果五类核心状态各自的主写入边界。
- **对外结果语义**：表示典型事件在最近结果、状态摘要、日志和 CLI 响应中的统一表达规则。

## 依赖与默认前提

- 当前单进程主循环、trigger 队列模型和命令协议仍然是系统正式运行方式，本次改动只在其内部收敛结构职责。
- 现有领域服务、应用协调逻辑和入口适配代码可以在不改变对外命令语义的前提下重新划分职责边界。
- 当前回归路径已经覆盖正式命令和典型运行流程，或可以在本次实现中补齐相应验证以证明行为稳定。
