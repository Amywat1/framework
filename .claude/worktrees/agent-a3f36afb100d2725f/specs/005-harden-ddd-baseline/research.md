# 研究记录：DDD 架构基线加固

## 决策 1：正式命令继续只有一个应用层正式入口，`status` 作为只读分支纳入同一入口

**Decision**：保留现有正式命令统一入口思路，由应用层统一受理 `start / stop / feedback / fault / fault clear / status`。其中写命令继续采用 `prepare -> submit -> main_loop_run -> finalize` 路径，`status` 采用 `prepare -> 只读查询 -> 响应生成` 路径，二者都不再形成适配层旁路。  
**Rationale**：
- 当前 `cli_command_adapter_execute_formal_line()` 已经能承接统一正式入口职责，直接在其上收敛风险最低。
- 规格已明确 `status` 必须进入统一正式入口，但保持只读且不进入 trigger 队列。
- 这样既保留了统一入口，又不强行把只读查询伪装成写事件。  
**Alternatives considered**：
- 让 `status` 继续旁路：会保留正式入口例外分支。
- 让 `status` 也入 trigger 队列：会把只读查询和运行时事件处理混在一起，增加无意义复杂度。

## 决策 2：领域服务改为返回领域事实，不再直接写最近结果或日志

**Decision**：`wash_session_state_machine`、`wash_execution_service`、`wait_timeout_service` 等领域服务不再直接调用 `runtime_event_recorder`，而是返回迁移结果、执行结果、等待结果、超时决议和原因信息，由应用层统一决定如何记录。  
**Rationale**：
- 当前领域服务直接依赖 `runtime_event_recorder`，形成领域反向依赖应用层的结构问题。
- 领域返回“事实”，应用层做“投影”，最符合本期“纠正依赖方向”的目标。
- 这能让领域规则变更不再牵连 CLI/日志/最近结果拼装逻辑。  
**Alternatives considered**：
- 保留领域内记录调用，只补充注释：不能真正纠正依赖方向。
- 将记录器下沉到领域层：会把 CLI/最近结果等外部语义继续拉入领域。

## 决策 3：去掉领域对整块 `system_context` 的直接依赖，改用窄化参数面

**Decision**：领域服务签名不再直接以 `system_context_t *` 作为唯一输入；改为接收自己真正需要的领域模型对象、端口对象以及必要的时间/标识信息。若某服务需要多个相关输入，则在相应领域头文件中定义只包含领域模型和端口的窄化输入结构。  
**Rationale**：
- 当前 `system_context` 透传让领域可以跨对象任意改写状态，主写入边界难以看清。
- 以领域模型和端口为边界定义输入，比在应用层自定义更多“万能运行时 DTO”更简单。
- 这能直接体现“领域服务到底需要什么”，并压缩越界写入空间。  
**Alternatives considered**：
- 继续传 `system_context` 但约定少用：约束不可验证。
- 在应用层新增大而全的运行时协调对象：容易演化成新的万能容器。

## 决策 4：应用层统一承担运行时结果投影与最近结果写入

**Decision**：最近结果、状态摘要、日志和 CLI 响应之间的语义映射统一由应用层完成；`runtime_event_recorder` 继续保留在应用协调层，作为“领域事实 -> 运行时记录”的集中投影工具。  
**Rationale**：
- 当前 `process_wash_trigger`、领域服务和适配层共同参与结果表达，容易产生重复和冲突。
- 规格已明确最近结果只允许统一主触发编排入口写入，`status` 不改写最近结果。
- 将投影统一收口到应用层，能保证对外表达一致且便于回归验证。  
**Alternatives considered**：
- 让适配层在生成响应时顺带刷新最近结果：会再次耦合 CLI 语义和运行时记录。
- 让每个领域服务自己记日志、写结果：会重新分散语义映射。

## 决策 5：`global_fault` 与最近结果的最终所有权留在统一主触发编排入口

**Decision**：`global_fault` 的设置与清除、最近结果的最终写入，统一保留在主触发编排入口；其他服务只返回事实、状态变化或决议结果，不直接写这两类跨对象状态。  
**Rationale**：
- 这两类状态都具有跨对象、对外可观测、需统一解释的特性，最适合保留在应用层编排边界。
- 规格已明确这两类状态的主写入边界，且 `status` 查询不能影响最近结果。
- 这样可直接降低状态定位成本，避免适配层和领域层顺带写跨对象状态。  
**Alternatives considered**：
- 单独新增故障协调器：超出本期最小范围。
- 允许多个应用服务按需写入：边界仍然不清晰。

## 决策 6：会话、执行、等待条件和超时决议继续分治，但结果拼装上收应用层

**Decision**：会话状态迁移仍由 `wash_session_state_machine` 承担；执行状态与等待条件仍由 `wash_execution_service` 承担；`wait_timeout_service` 只产出超时决议；跨对象编排和结果投影上收应用层。  
**Rationale**：
- 当前代码已经存在按状态类别拆分的基础，只是被 `system_context` 透传和直接记录逻辑削弱了边界。
- 延续既有服务职责比新增新中心更稳定，且符合“保持现有运行模型”的要求。
- 这样改动集中在签名、返回值和编排逻辑，利于以最小范围完成重构。  
**Alternatives considered**：
- 新建统一运行时管理器：复杂度过高。
- 允许 timeout service 直接接管执行/等待写入：会再次造成职责重叠。

## 决策 7：兼容性 use case 保留文件，但不再承担正式主路径地位

**Decision**：`start_wash_cycle.c`、`stop_wash_cycle.c`、`acknowledge_fault.c` 可以保留为兼容性包装或测试辅助，但正式生产路径继续集中在统一正式入口和主触发编排入口。  
**Rationale**：
- 这样能兼容现有测试或内部调用，同时避免它们继续代表第二套正式写入口。
- 文件保留但角色降级，符合“不要为形式纯度做目录级大搬迁”的要求。
- 可把变更聚焦在职责纠偏，而不是接口删改。  
**Alternatives considered**：
- 直接删除这些文件：实现期可能带来不必要的测试改动扩散。
- 继续允许其作为生产主路径：会保留结构歧义。
