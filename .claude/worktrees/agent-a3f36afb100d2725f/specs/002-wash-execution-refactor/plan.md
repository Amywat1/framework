**说明**：本计划基于当前已澄清的规格、项目宪章和现有仓库结构生成，目标是在保留 C 语言、Linux、DDD 分层和六边形边界的前提下，将现有串行洗车主路径升级为会话驱动、执行驱动、事件驱动与超时驱动的模型。

## 概要

本次工作聚焦于“如何推进一次洗车任务”，而不是重新定义完整洗车工艺。计划中的最小可行实现是在现有 `wash_cycle`、`wash_cycle_coordinator`、`start_wash_cycle`、`main_loop` 和配置/日志适配器基础上，引入“洗车会话 + 单次执行 + 触发事件 + 超时等待 + 程序快照”五类核心对象，并补充统一状态观测入口，形成显式、可中断、可观测、可扩展的推进模型。  
设计上继续采用“领域模型 + 应用编排 + 适配器边界 + Linux 事件循环”的组合：领域层承载会话状态、执行状态、优先级、等待重试与结果规则；应用层负责编排启动、停止、推进、超时、状态查询和结果归档；平台与适配器层负责命令输入、设备反馈、超时发现、状态输出和日志输出。

## 假设、问题与权衡

**假设**：现有命令入口、设备反馈入口和日志出口会继续存在；程序配置仓储能够在启动时返回一份稳定的程序内容；当前主循环允许引入“事件发现”和“超时发现”而不要求重建全部平台抽象。  
**未决问题**：无  
**被拒绝的方案**：  
1. 继续沿用单个串行入口函数完成整套洗车流程：被拒绝，因为不满足事件/超时驱动和状态分离要求。  
2. 将会话与执行压缩成同一个状态对象：被拒绝，因为会掩盖“单次执行结束但会话继续”的后续决策路径。  
3. 引入多会话排队能力：被拒绝，因为当前规格已明确单机单车位、运行中启动必须拒绝。  
4. 通过轮询逻辑直接推进业务状态：被拒绝，因为规格已明确固定周期检查只能用于发现事件或发现超时，不能直接作为业务推进依据。  
**最小可行方案**：在现有洗车周期域模型之上新增洗车会话对象、单次执行对象、等待条件与程序快照；由 Linux 主循环统一收集启动、停止、设备反馈、故障和超时五类正式触发源，再由应用层调用单次“推进”入口驱动状态迁移；其他业务事件仅保留类型位，不在本次实现具体推进分支。  
**范围边界**：本次计划只触及 `domain/model`、`domain/services`、`application/coordinators`、`application/use_cases`、`platform/linux`、相关适配器与测试；不改变完整程序配置格式的业务外延，不新增云协议，不扩大维护模式能力。

## 嵌入式 C 安全计划

**是否适用**：是  
**命名与分层**：文件/函数/变量使用 `snake_case`；宏全大写；类型以 `_t` 结尾；全局变量前缀 `g_`；静态变量前缀 `s_`；头文件只放声明；继续保持 `domain / application / adapters / platform / main` 分层。  
**格式与常量**：统一 4 空格缩进；禁用 Tab；所有控制语句带花括号；状态、结果、超时、重试次数、优先级全部使用具名常量或枚举。  
**错误策略**：启动校验失败、设备反馈无效、重复启动、迟到事件、状态不匹配、超时收敛失败都必须返回显式结果或错误码；安全类故障优先级最高。  
**等待策略**：所有等待型步骤都必须定义等待条件、等待起点、超时阈值、最大重试次数和重试耗尽后的单一收敛结果；若业务不允许重试，则必须显式定义最大重试次数为 `0`。  
**中断策略**：若底层仍使用中断或轻量唤醒，只允许事件置位或唤醒，不在 ISR 中完成会话推进、日志写入或动态内存分配。  
**并发与内存安全**：会话/执行共享状态由单线程事件循环或受控锁保护；程序快照必须在生命周期内保持只读；所有外部输入都要做空指针、边界和状态匹配校验。  
**状态机**：洗车会话、单次执行、必要时等待型步骤都采用显式状态枚举和迁移规则；必须定义默认状态、终态、拒绝路径和唯一结果收敛规则。  
**注释风格**：中文注释；公开头文件、状态迁移、推进入口、适配器契约使用 Doxygen 风格。

## 复杂度跟踪

| 违规项 | 为什么必须这样做 | 为什么更简单方案不可行 |
|--------|------------------|------------------------|
| 引入“会话 + 单次执行”双层运行时对象 | 规格明确要求分离会话级与执行级状态，并支持“执行结束后由会话决定后续” | 单层对象无法清晰表达等待、结果回传和后续决策边界 |
| 在 Linux 主循环中增加超时发现与事件汇聚 | 规格要求业务推进只能由事件或超时驱动 | 直接继续串行调用会破坏新执行模型，纯业务层也无法自行发现超时 |
| 为启动拒绝、重复启动、迟到事件增加显式结果契约 | 这些路径已在澄清中成为必须验证的规则 | 如果只写内部判断，不定义契约和记录要求，后续任务拆解与测试边界会模糊 |
