# 快速开始：洗车执行模型重构

## 1. 目标

本指南用于帮助开发与评审人员验证新的执行模型是否成立：  
1. 洗车任务由“会话 + 单次执行”组织；  
2. 主流程只由事件或超时推进；  
3. 停止、故障、超时、重复启动和无效启动都进入明确路径；  
4. 运行中的程序数据不会被外部修改直接影响；  
5. 维护人员可通过统一状态查询入口观察当前会话、当前执行与最近原因。

## 2. 前置条件

- Linux 开发环境可用
- C17 编译器与 CMake 可用
- 现有项目源码、测试目录与仿真驱动可用
- 已准备至少一份可用程序配置，以及一份故意无效的程序配置

## 3. 推荐运行时目录

```text
configs/
|-- programs/
|   |-- standard_wash.json
|   `-- invalid_program.json
`-- system_limits.json

runtime/
|-- logs/
`-- state/
```

## 4. 构建

```bash
cmake -S . -B build
cmake --build build
ctest --test-dir build --output-on-failure
```

## 5. 验证场景

### 场景 A：正常推进闭环

1. 启动主控与仿真环境。
2. 发送一次启动请求，使用有效程序配置。
3. 逐步注入阶段所需设备反馈。
4. 观察以下结果：

- 创建一个新的洗车会话
- 会话进入运行中
- 每次设备反馈只推进当前单次执行一步
- 在没有事件也没有超时时，会话与执行状态保持不变
- 最终形成单一完成结果

### 场景 B：运行中重复启动被拒绝

1. 在场景 A 的运行中阶段，再次发送启动请求。
2. 观察以下结果：

- 当前会话保持不变
- 不创建并行会话
- 生成重复启动拒绝记录

### 场景 C：启动前程序数据校验失败

1. 使用无效程序配置发送启动请求。
2. 观察以下结果：

- 请求被拒绝
- 不创建新会话
- 生成“启动前校验失败”记录

### 场景 D：超时驱动收敛

1. 启动一次有效会话。
2. 在某个等待阶段不注入反馈，直到等待超时。
3. 观察以下结果：

- 超时被识别为正式触发源
- 若允许重试，则等待按预定义最大重试次数重新建立
- 当前执行进入明确的超时结果
- 会话根据超时策略继续、结束当前执行后决策，或直接中止

### 场景 E：竞争事件优先级

1. 在同一执行窗口内构造故障、停止、超时与设备反馈竞争。
2. 观察以下结果：

- 系统只收敛到一个处理结果
- 优先级遵循“故障 > 停止 > 超时 > 设备反馈”
- 迟到事件和重复事件被忽略并记录

### 场景 F：统一状态查询

1. 启动一次有效会话并推进到某个等待阶段。
2. 调用统一状态查询入口。
3. 观察以下结果：

- 能直接读取当前会话状态
- 能直接读取当前执行状态与阶段标识
- 能直接读取最近一次状态迁移原因

## 6. 最低测试覆盖建议

- 单元测试：会话状态迁移、执行状态迁移、竞争事件优先级、启动拒绝规则
- 集成测试：主循环事件汇聚、超时发现、执行结果回传会话
- 契约测试：启动/停止命令语义、结果事件语义、程序快照冻结语义
- 仿真测试：正常推进、重复启动、无效启动、超时、故障竞争
