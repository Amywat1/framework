# Contract：调度器退出与失败语义

## 目的

约束平台调度器在退出、排空与失败场景下的正式行为，确保控制器不会进入无限排空、静默停机或无法解释的中间状态。

## 参与者

- `controller_scheduler`：平台层公开抽象
- Linux 私有实现：负责退出标志、排空计数和错误归因
- `main.c`：只负责启动与接收最终退出结果
- `process_formal_command_execute()`：不得自行决定排空次数或退出策略

## 契约规则

1. 调度器运行状态只允许在 `initialized -> running -> draining -> stopped/failed` 定义域内迁移。
2. 收到退出事件后，调度器必须按配置进入“有界排空”或“直接退出”之一，不得无限继续推进。
3. 有界排空的上界必须由显式配置决定，达到上界后必须收敛到 `stopped` 或 `failed`。
4. 周期源失效、事件源关闭、输入读取失败、`eventfd` 创建/注册/读写失败、触发队列满和单拍推进失败都必须产生单一明确的错误结论。
5. 最近错误原因必须保留最后一次正式失败结论，供状态查询、日志和测试读取。

## 可验证结果

- 退出请求到来后，调度器只会进入 `draining` 或 `stopped`
- 排空过程有明确上界，不出现无限 `drain_runtime()`
- 失败路径都能得到单一明确的退出或错误结论

## 失败语义

- 配置非法或调度器初始化失败：直接返回失败，不进入 `running`
- 周期源或等待原语失效：进入 `failed`
- 排空期发生不可恢复错误：进入 `failed`
- 直接退出策略命中：从 `running` 进入 `stopped`
