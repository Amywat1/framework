# 实施计划：洗车主控运行骨架

**分支**：`003-wash-controller-runtime` | **日期**：2026-05-06 | **规格**：[spec.md](C:/Users/HUAWEI/Desktop/Code-M8/specs/003-wash-controller-runtime/spec.md)  
**输入**：来自 `/specs/003-wash-controller-runtime/spec.md` 的功能规格

**说明**：本计划基于当前规格、项目宪章和已有执行内核生成，目标是在保持 C17、Linux、DDD 分层和显式状态机方向不变的前提下，把当前“可测试的单次执行内核”接成“可长期运行的主控骨架”，并以标准输入命令、超时推进、启动前预检、状态查询和关键日志作为本阶段最小闭环；其中 `wash_controller` 是本期唯一正式运行入口，`wash_cli` 仅保留兼容/调试定位；全局故障只保留主控骨架级最小阻断职责，不扩展为通用故障管理子系统。

## 概要

本次工作不重新定义洗车工艺，也不引入多进程、云端或队列能力。最小可行方案是在现有 `system_context`、洗车会话/执行模型、主循环和状态查询能力之上，补齐四个核心能力：  
1. 让主控进程常驻，持续保留 `system_context` 并连续接单；  
2. 把 `stdin` 行文本命令收敛为统一的正式输入通道；  
3. 让主循环持续处理“输入事件 + 超时事件 + 优先级竞争”，而不是一次调用后退出；  
4. 在启动入口补预检、在状态查询中补主控骨架级全局故障/最近原因等可观测信息。  

设计上继续沿用当前单进程、分层清晰的控制器结构：领域层负责状态机、等待和故障规则；应用层负责编排启动/停止/反馈/故障/查询；适配器层负责 `stdin/stdout` 命令协议；平台层负责常驻循环、时间推进和事件汇聚。

## 假设、问题与权衡

**假设**：现有 `wash_session`、`wash_execution`、`wait_condition`、超时规则和优先级比较逻辑仍可作为运行时内核复用；控制器主线程可长期持有 `system_context`；当前代码基线允许在单进程中用阻塞读或轮询读方式接收 `stdin`。  
**未决问题**：无。当前计划所需的输入协议、错误处理和全局故障语义已在规格中明确。  
**被拒绝的方案**：  
1. 继续以一次性 `main_loop_run(..., program_id)` 驱动整单流程：被拒绝，因为不满足常驻主控和正式输入通道要求。  
2. 直接把真实运行入口做成 socket/IPC：被拒绝，因为规格已固定本阶段正式输入通道为 `stdin` 行命令。  
3. 把空闲时 `fault` 当作普通忽略事件：被拒绝，因为规格已明确必须沉淀为全局故障并阻断后续 `start`。  
4. 在每单结束后重建 `system_context`：被拒绝，因为会破坏常驻主控和连续接单目标。  
**最小可行方案**：保留单进程和当前执行内核，增加一个长期运行的命令处理循环；由 `wash_controller` 通过 `stdin` 解析 `start / stop / feedback / fault / status` 命令，转换为统一的触发/查询请求；主循环持续发现并处理输入事件与超时事件；显式复用并在必要时收口现有 `wait_timeout_service` 与 `trigger_priority_service`；启动路径接入预检；状态查询补足全局故障与最近原因；全局故障在本期仅提供记录、阻断 `start` 和 `fault clear` 清除三项最小能力。  
**范围边界**：本次计划只覆盖运行骨架层面的接线、协议、观测和测试，不改变洗车阶段程序模型，不引入多会话排队，不实现完整故障中心，也不把本期全局故障扩展为通用故障分类、分级处置或恢复编排能力。

## 嵌入式 C 安全计划

**是否适用**：是  
**命名与分层**：文件/函数/变量统一使用 `snake_case`；宏全大写；类型以 `_t` 结尾；全局变量前缀 `g_`；静态变量前缀 `s_`；头文件只放声明；继续遵守 `domain / application / adapters / platform / main` 分层。  
**格式与常量**：统一 4 空格缩进；禁用 Tab；控制语句必须带花括号；命令类型、响应结果、全局故障状态和固定字符串长度使用具名常量或枚举。  
**错误策略**：未知命令、参数缺失、预检失败、全局故障阻断、运行中重复启动、无活动任务的 stop/feedback、状态不匹配输入都必须返回显式结果并记录最近原因。  
**等待策略**：继续复用现有等待条件和超时规则；控制器运行骨架不得把等待退化成无界阻塞业务状态，超时仍必须由主循环持续发现并推进。  
**中断策略**：本阶段不新增 ISR；若未来底层有中断唤醒，本骨架只假设其将事实转换为轻量输入，不在中断上下文内推进状态或写日志。  
**并发与内存安全**：默认单线程主控循环，不引入共享多线程状态；所有命令解析都必须做参数个数、空指针和缓冲区长度检查；全局故障和最近原因字段必须显式初始化与重置。  
**状态机**：会话状态、执行状态、等待状态和全局故障阻断规则必须保持显式；`status` 查询只读，不得产生状态迁移。  
**注释风格**：中文注释；所有新增或修改的公开头文件接口必须补中文 Doxygen 注释，覆盖用途、参数、返回值和必要约束。

## 复杂度跟踪

| 违规项 | 为什么必须这样做 | 为什么更简单方案不可行 |
|--------|------------------|------------------------|
| 在现有执行内核外再加一层常驻主控循环 | 规格要求同一进程连续接单并长期保留上下文 | 仅保留当前一次性调用无法满足“启动后不退出” |
| 在 `system_context` 中补主控级全局故障信息 | 规格要求空闲 fault 形成可查询、可阻断启动、可显式清除的最小主控级状态 | 只靠任务级状态无法表达“无活动任务但存在故障”，但又不能上升为完整故障中心 |
| 定义 `stdin/stdout` 单行命令契约 | 正式输入通道和单行响应是本次核心外部接口 | 若只靠测试注入事件，主控骨架无法形成真实运行闭环 |
