# 研究记录：洗车主控运行骨架

## 决策 1：采用单进程常驻主控 + 长生命周期 `system_context`

**Decision**：主控进程启动后不退出，在同一进程内长期持有一个 `system_context`，只在每单切换时重置任务级状态，不重建整个运行时骨架。

**Rationale**：
- 规格明确要求“启动后持续保留 system_context”并连续处理多次任务。
- 当前代码已经以 `system_context` 聚合会话、执行、等待和端口，直接复用成本最低。
- 单进程长生命周期模型最容易验证“上一单结束后继续接下一单”和“状态查询随时可读”。

**Alternatives considered**：
- 每单重建一次上下文：实现简单，但不满足常驻主控目标。
- 将上下文拆成多进程共享：超出本次范围，也会显著增加复杂度。

## 决策 2：正式输入通道采用 `stdin` 行文本协议

**Decision**：把主控正式输入固定为 `stdin` 单行命令，统一支持 `start <program_id>`、`stop`、`feedback <signal_code>`、`fault <fault_code> <reason>`、`fault clear` 和 `status`。

**Rationale**：
- 这是规格已明确的输入形态，且最接近当前仓库已有 CLI 入口。
- 行文本协议便于仿真、自动化测试和联调，不需要额外依赖。
- 命令解析后可无损转换为统一的触发/查询请求，与当前领域模型耦合最小。

**Alternatives considered**：
- 继续使用命令行参数单次调用：无法形成常驻会话。
- 直接引入 socket/IPC：不是当前规格要求的最小闭环。

## 决策 3：每条命令都返回一条单行 `stdout` 结果

**Decision**：控制器对每一条输入命令都返回一条可判断的单行文本结果；`status` 返回当前状态单行摘要；错误命令返回单行错误并继续运行。

**Rationale**：
- 规格已经把“单行响应”定义为正式外部契约。
- 单行结果最利于脚本驱动测试和运行时人工观察。
- 它允许把“受理 / 拒绝 / 忽略 / 当前状态”统一成稳定协议，而不暴露内部结构。

**Alternatives considered**：
- 只返回 exit code：无法满足常驻进程逐命令交互。
- 输出多行详情：信息更全，但不符合当前协议约束，也会增加解析复杂度。

## 决策 4：空闲 `fault` 提升为主控级全局故障，并阻断后续 `start`

**Decision**：当没有活动任务时收到 `fault <fault_code> <reason>`，主控记录全局故障状态和最近原因；在全局故障未被 `fault clear` 清除前，所有 `start` 都必须被拒绝。

**Rationale**：
- 这是规格澄清后的刚性规则，必须在设计阶段明确到数据模型和契约层。
- 该规则将“设备已知异常但当前无任务”的状态从任务级语义中分离出来。
- 它能为后续完整故障中心预留稳定边界，同时不需要本次实现完整故障树。

**Alternatives considered**：
- 空闲 `fault` 直接忽略：违反规格。
- 空闲 `fault` 临时挂在最近原因字段：无法稳定阻断后续启动，也不利于状态查询。

## 决策 5：启动前预检作为 `start` 受理门槛，而不是启动后的补救路径

**Decision**：`start` 命令先执行资源/传感器预检；预检失败时直接拒绝本次启动、记录原因并保持主控继续可用，不创建任务后再终止。

**Rationale**：
- 规格明确要求“启动前检查”和“失败时拒绝启动并记录原因”。
- 现有 `precheck_service` 已存在，接入成本低。
- 先判定后创建可避免无效会话污染日志、状态和后续流程。

**Alternatives considered**：
- 先创建会话再失败中止：会制造多余状态迁移和假任务。
- 完全跳过预检：不满足启动边界要求。

## 决策 6：主循环继续负责“发现并挑选”事件，不直接绕开优先级与超时规则

**Decision**：主循环长期运行并持续汇聚两类事实来源：正式输入命令转换后的触发事件，以及等待条件派生出的超时事件；随后继续复用统一优先级规则收敛到单一处理结果。

**Rationale**：
- 规格要求“事件驱动 + 超时驱动”，且同一窗口只能收敛到一个明确结果。
- 现有 `main_loop`、`trigger_priority_service` 和 `wait_timeout_service` 已经具备雏形。
- 这条路径能把主控骨架升级控制在最小改动范围内，而不是重写整套调度器。

**Alternatives considered**：
- 输入一来就直接执行对应用例，超时另起旁路：会破坏统一竞争规则。
- 用固定轮询步骤直接推进业务状态：与当前规格冲突。

## 决策 7：状态查询最小扩展为“任务状态 + 等待状态 + 最近原因 + 全局故障”

**Decision**：在现有 `query_wash_session_status` 基础上，扩展状态视图以覆盖：是否有活动任务、会话状态、执行状态、当前等待条件、最近原因、是否存在全局故障、最近全局故障原因。

**Rationale**：
- 规格明确把状态查询和日志增强作为本阶段交付物。
- 当前状态查询已具备基础字段，适合增量扩展。
- 该视图既支撑 `status` 命令，也支撑测试和后续联调。

**Alternatives considered**：
- 只保留现有会话/执行状态：无法表达全局故障和等待边界。
- 直接暴露内部结构体全文：耦合过高，不适合作为正式外部契约。
