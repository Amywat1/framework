# 功能规格：主控 Runtime/Bootstrap 正式入口

**功能分支**：`010-controller-runtime-bootstrap`  
**创建日期**：2026-05-11  
**状态**：草稿  
**输入**：用户描述：“把主控组合根和启动/停机闭环做完整，把分散在主程序、测试辅助和适配器初始化里的装配逻辑收口成一套正式运行入口；统一 system_context 获取、端口装配、scheduler 创建、运行与销毁流程；显式返回启动失败；固定顺序回收资源；复用同一套装配流程；保持 system_context 只承载组合根运行期状态；新增更上层 runtime/bootstrap 入口；采用 create/run/destroy；严格区分 runtime owned 和 caller owned 资源；避免平台细节无控制泄漏。”  

## Clarifications

### Session 2026-05-11

- Q: 同一进程内正式 runtime 实例的并存数量如何定义？ → A: 同一进程同一时刻只允许 1 个正式 runtime 实例。
- Q: `run` 返回后同一 runtime 实例是否允许再次运行？ → A: `run` 返回后实例进入终态，只允许 `destroy`，不允许再次 `run`。
- Q: `run` 的返回结果应如何区分正常退出与错误退出？ → A: 正常退出算成功；只有错误路径算失败。

## 澄清、假设与权衡（必填）

- **假设**：当前主控仍采用单进程、阻塞式运行模型；正式运行入口本期只需要覆盖 `create/run/destroy`，不引入异步 `start/stop` 语义。
- **假设**：`run` 是一次性进入运行的动作；无论其因正常退出还是失败退出而返回，原 runtime 实例都不得再次进入 `run`，只能进入销毁路径。
- **假设**：`run` 的返回结果需要区分正常退出与错误退出；正常完成退出闭环应返回成功，只有运行期错误、生命周期误用或内部故障才返回失败。
- **假设**：同一进程同一时刻只允许存在 1 个正式 runtime 实例；若已有实例处于已创建未销毁状态，新的正式实例创建请求必须显式失败。
- **假设**：驱动上下文、测试桩对象、标准输入输出绑定和外部配置源由调用方提供，正式 runtime 只负责绑定和解除绑定，不负责销毁这些调用方拥有的对象。
- **假设**：程序仓储和事件日志属于启动闭环中的必备装配项；缺失、路径错误或初始化失败都应在运行开始前暴露为显式失败。
- **未决问题**：不适用。
- **解释选择**：本期优先把正式运行入口放在当前主控组合根之上的协调层，接受其为当前主控可执行骨架服务的工程务实取舍，但要求公开接口只暴露运行实例语义，不暴露平台实现细节。
- **不在范围内**：不调整洗车工步模型，不修改 JSON 工艺规则语义，不重设计故障策略，不新增真机驱动能力，不扩展界面层。

## 用户场景与测试（必填）

### 用户故事 1 - 统一创建并运行主控实例（优先级：P1）

作为主控启动入口的拥有者，我希望通过一套正式 runtime 入口完成主控实例创建与运行，这样主程序不再手工拼装 `system_context`、端口和 scheduler，主控运行边界只有一条正式路径。

**为什么是这个优先级**：这是后续所有故障闭环、平台接入和资源治理的前提；如果正式入口不唯一，后续新增能力仍会继续分散到多个入口。

**独立验证方式**：使用正式 runtime 入口创建一个主控实例并运行，确认创建结束时实例已处于“可运行但未运行”状态，且 `system_context`、必需端口、程序仓储、事件日志和 scheduler 均已就绪；运行阶段不再进行额外装配。

**验收场景**：
1. **给定** 调用方已提供必需的驱动绑定、配置路径和运行参数，**当** 创建正式 runtime 实例并进入运行，**则** 主控应只经过一套正式创建和运行路径完成启动。
2. **给定** 正式 runtime 实例已创建但尚未运行，**当** 读取其状态，**则** 应可判定其已具备运行条件且未开始执行主循环。

---

### 用户故事 2 - 启动失败显式返回并完整回滚（优先级：P1）

作为主控运行实例的拥有者，我希望启动阶段的缺失配置、路径错误、端口未装配或 scheduler 创建失败都能显式返回，并自动回滚已占用资源，这样不会留下半初始化对象或悬空绑定。

**为什么是这个优先级**：如果启动失败不能统一回滚，后续故障处理和重复启动都无法建立稳定前提，系统会在最基础的生命周期边界上失控。

**独立验证方式**：分别制造仓储初始化失败、日志初始化失败、scheduler 创建失败等场景，验证创建调用返回明确失败，并且失败后能立即重新创建新实例。

**验收场景**：
1. **给定** 创建阶段出现任一必备资源装配失败，**当** 调用正式 runtime 创建入口，**则** 应返回明确失败结果且不留下已占用的运行实例资源。
2. **给定** 上一次创建在中间步骤失败，**当** 立即再次创建新实例，**则** 新实例应可正常获取资源并完成创建。

---

### 用户故事 3 - 测试与正式入口共用同一套装配闭环（优先级：P2）

作为测试和适配层维护者，我希望测试辅助和正式可执行入口复用同一套 runtime 装配流程，只通过配置注入替换驱动、路径、stdio 和测试桩，这样装配边界不会在测试侧重新分叉。

**为什么是这个优先级**：这能防止测试代码继续复制正式装配逻辑，把本轮收紧的生命周期边界长期固定下来。

**独立验证方式**：让测试辅助通过正式 runtime 装配入口创建实例，仅替换测试驱动和路径注入，确认不再复制主程序中的手工装配步骤。

**验收场景**：
1. **给定** 正式入口与测试入口使用不同的驱动和路径注入，**当** 它们各自创建 runtime 实例，**则** 两者应共享同一套创建与销毁闭环，仅在注入配置上存在差异。

### 边界场景

- 当调用方缺失必需装配输入时，创建必须立即失败，而不是生成可部分使用的运行实例。
- 当同一进程中已有一个正式 runtime 实例尚未销毁时，新的正式 runtime 创建必须显式失败，而不是并存运行或抢占原实例。
- 当 `run` 已经返回时，原 runtime 实例必须进入终态，不得再次运行，也不得重新回到“已创建未运行”状态。
- 当 `run` 因正常退出条件返回时，系统必须将其判定为成功完成运行闭环，而不是错误失败。
- 当日志路径、配置路径或 scheduler 创建失败时，系统必须回滚已创建的 runtime owned 资源，并解除已建立的绑定。
- 当 `destroy` 作用于空指针、部分创建成功对象或已销毁对象时，系统必须安全返回，不得要求外层分支判断。
- 当实例已销毁后，旧句柄不得继续表现为有效的 scheduler 绑定或已占用的运行实例资源。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须提供一套正式 runtime/bootstrap 入口，用于表示一个主控运行实例的创建、运行和销毁闭环。
- **FR-002**：系统必须将正式运行实例的生命周期固定为 `create/run/destroy`，本期不得引入高于现状的异步 `start/stop` 语义。
- **FR-003**：系统必须在 `create` 成功返回前完成 `system_context` 获取、必需端口装配、程序仓储装配、事件日志装配和 scheduler 创建，使实例处于“可运行但未运行”状态；`create` 成功返回时不得再存在需要由 `run` 补做的正式装配步骤。
- **FR-004**：系统必须保证 `run` 只负责进入正式运行，不再在运行开始时补做装配、绑定或资源创建。
- **FR-004A**：系统必须保证 `run` 是一次性运行入口；任一 runtime 实例在 `run` 返回后都必须进入不可再次运行的终态，只允许执行销毁。
- **FR-004B**：系统必须将 `run` 的返回结果区分为“正常退出成功”与“错误退出失败”；正常退出不得被表示为失败结果。
- **FR-005**：系统必须对创建阶段的配置缺失、路径错误、端口未绑定、适配器初始化失败和 scheduler 创建失败返回明确结果，而不是静默降级或延后到运行期暴露。
- **FR-006**：系统必须为任一创建失败场景执行统一的逆序回滚，释放已创建的 runtime owned 资源，并解除已建立的运行时绑定。
- **FR-007**：系统必须将资源所有权显式区分为 `runtime owned` 与 `caller owned`，其中 runtime 只销毁自己创建的资源，不得销毁调用方拥有的驱动上下文、测试桩对象或外部配置源。
- **FR-008**：系统必须定义统一销毁顺序，并允许 `destroy` 处理空对象、部分创建成功对象和已销毁对象，而不要求调用方识别中间态。
- **FR-009**：系统必须保证正式可执行入口只通过这套 runtime 入口启动主控，不再直接调用底层装配步骤拼装实例。
- **FR-010**：系统必须允许测试入口复用同一套正式装配流程，并通过注入方式替换驱动、路径、stdio、调度参数和测试桩实现。
- **FR-011**：系统必须在实例销毁完成后，确保旧 runtime 相关句柄不再表现为有效的 scheduler 绑定或可继续占用的运行实例资源。
- **FR-012**：系统必须将正式运行入口的状态表达限制在运行实例生命周期和装配结果范围内，不得借机扩大 `system_context` 的业务职责。
- **FR-013**：系统若提供只读运行态观测接口，其范围必须限制在生命周期状态、装配结果和失败回滚验证所必需的最小集合内，不得借机扩展为新的运行时管理面或平台细节透传面。
- **FR-014**：系统必须保证同一进程同一时刻只允许存在 1 个正式 runtime 实例；当已有正式实例处于已创建未销毁状态时，新的正式实例创建请求必须返回明确失败。

### 范围与简化要求（必填）

- **SS-001**：系统必须只实现主控实例生命周期、装配边界、启动失败路径和资源回收闭环所必需的最小能力。
- **SS-002**：系统不得在本轮中调整洗车业务流程语义、工步段模型、JSON 结构或故障策略设计。
- **SS-003**：系统的变更必须集中在正式 runtime 入口、装配闭环、适配器初始化返回链路以及正式入口与测试辅助的装配收口范围内。

### 关键实体（若功能涉及数据则必填）

- **Controller Runtime**：表示一个正式主控运行实例，负责承载创建后可运行、运行中和销毁后的生命周期状态，并拥有 runtime owned 资源。
- **Runtime Create Request**：描述创建正式主控实例所需的注入信息，包括调用方提供的绑定输入、运行参数和外部路径等，用于驱动统一装配流程。
- **Runtime Ownership Boundary**：描述运行实例内部哪些资源由 runtime 负责创建和销毁，哪些资源仅由调用方提供并保持所有权不变。
- **Startup Result**：描述创建阶段的结果，包括成功进入可运行状态，或因必备装配失败而返回明确失败并完成回滚。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：在覆盖的启动失败场景中，100% 都能在进入正式运行前返回明确失败结果，而不是在运行后才暴露装配问题。
- **SC-002**：在覆盖的创建失败和正常销毁场景中，100% 都能在同一进程内立即重新创建新的主控实例，不出现残留的实例占用或 scheduler 绑定。
- **SC-003**：正式可执行入口的主控启动路径收敛为一套正式 runtime 入口，不再保留分散的手工装配主路径。
- **SC-004**：测试入口与正式入口共用同一套装配编排，审计到的差异只存在于驱动、路径、stdio、调度参数或测试桩注入层。
- **SC-005**：在覆盖的并发创建场景中，100% 都能阻止同一进程内出现多个正式 runtime 实例并存，且第二次创建返回明确失败结果。
- **SC-006**：在覆盖的运行结束场景中，100% 都能阻止同一 runtime 实例在 `run` 返回后再次进入运行，并且重复运行请求返回明确失败。
- **SC-007**：在覆盖的正常退出场景中，100% 都能将 `run` 返回判定为成功，而不是误记为失败路径。

## 验证方式（必填）

- **V-001**：验证正式 runtime 入口能将主控实例创建到“可运行但未运行”状态，且创建成功时 `system_context`、必需端口、程序仓储、事件日志和 scheduler 已全部就绪，并由独立运行调用进入正式运行。
- **V-002**：验证仓储初始化失败、日志初始化失败、scheduler 创建失败和缺失必需绑定时的显式失败与统一回滚。
- **V-003**：验证 `destroy` 对空对象、部分创建对象和已销毁对象的安全处理，以及销毁后旧绑定失效。
- **V-004**：验证测试辅助与正式入口复用同一套装配闭环，仅通过注入替换测试驱动和运行参数。
- **V-005**：验证同一进程内已有一个正式 runtime 实例尚未销毁时，第二次正式创建会显式失败，且不会破坏已存在实例。
- **V-006**：验证 `run` 返回后的 runtime 实例不能再次进入运行，且仅允许进入销毁路径。
- **V-007**：验证 `run` 在正常退出条件下返回成功结果，而在运行期错误或生命周期误用时返回明确失败。

## 依赖与默认前提

- 调用方能够提供正式 runtime 创建所需的驱动绑定输入、运行参数和外部路径。
- 当前主控仍沿用阻塞式运行模型，因此运行实例在 `run` 返回前持续持有其 runtime owned 资源。
- 程序仓储和事件日志适配器在本轮中被视为启动闭环中的正式依赖，创建阶段必须能验证它们可被成功装配。
