# 功能规格：主控骨架边界收紧

**功能分支**：`007-controller-boundary-hardening`  
**创建日期**：2026-05-08  
**状态**：草稿  
**输入**：用户描述：“当前这轮改造的目标，不是重做流程模型，也不是继续加洗车工艺能力，而是把主控骨架的边界收紧，让后续扩展时不失控。”

## Clarifications

### Session 2026-05-08

- Q: 会话结束后的唯一最终落点应该放在哪里？ → A: `wash_session.final_session_result` 是唯一最终落点；`last_result_code` 只做最近一次对外投影
- Q: 发现跨层直写时应按什么规则处理？ → A: 视为实现违规：必须改成正式入口/拥有者写入，并由测试防回退
- Q: 外部适配层是否可以直接写触发队列？ → A: 不可以；适配层只能提交事实/请求，触发队列是主循环内部机制
- Q: “正式提交入口” 的规范化对象是谁？ → A: `process_formal_command` 只负责 CLI 正式命令；其他适配层必须走各自明确的应用入口，统一规则是都不能直接碰队列
- Q: 兼容性用例入口应保留什么定位？ → A: 全部删除，只保留正式命令入口

## 澄清、假设与权衡（必填）

- **假设**：当前主控仍以单进程、单主循环方式运行；现有工步段模型、程序配置和对外控制语义在本期继续有效，本期只重构运行期边界与状态约束。
- **未决问题**：不适用
- **解释选择**：本规格将“边界收紧”解释为限制共享状态的写入口、收缩服务依赖面、明确运行期对象职责边界，并用回归验证约束这些边界不会回退；不把它解释为流程模型重做或业务能力扩张。
- **不在范围内**：不改工步段模型语义；不改 JSON 工艺规则本身；不做程序在线编辑或保存；不做真机驱动扩展；不做界面层扩展。

## 用户场景与测试（必填）

### 用户故事 1 - 总上下文只承载共享运行事实（优先级：P1）

作为主控架构维护者，我希望 `system_context_t` 被定义为运行期总容器而不是任何代码都能顺手改的公共状态，这样后续新增功能时，共享状态不会继续无边界膨胀。

**为什么是这个优先级**：如果总上下文的定位不先收紧，后续所有扩展都会继续把临时数据和局部决策塞进同一个容器，复杂度会持续失控。

**独立验证方式**：检查总上下文中保留的字段是否都属于真正跨服务共享、且需要主控统一解释的运行事实；确认局部临时数据不再默认进入总上下文。

**验收场景**：

1. **给定** 某服务只在本次调用期间临时使用一组中间数据，**当** 维护者检查该数据归属时，**则** 该数据不会被放入 `system_context_t` 作为默认共享状态。
2. **给定** 主控需要对外解释当前运行事实，**当** 维护者检查 `system_context_t` 中的共享字段时，**则** 每个字段都能说明其共享必要性和最终解释责任。
3. **给定** 新增一项后续扩展能力，**当** 评审其数据落点时，**则** 能先区分哪些数据属于主控共享状态，哪些只属于单个服务或单次处理过程。

---

### 用户故事 2 - 关键运行状态只允许从正式入口改写（优先级：P1）

作为排查运行状态问题的维护者，我希望全局故障、最近结果、触发队列、当前时间等关键运行状态只允许从少数固定入口改写，这样状态变化来源始终可追踪、可解释。

**为什么是这个优先级**：主控最危险的问题不是流程不跑，而是状态能跑通但解释不一致；关键状态若被多处直接改写，后续故障定位会迅速失真。

**独立验证方式**：逐类检查关键运行状态，确认每类状态的写入口数量受控、写入时机明确、写入后的对外表现可说明。

**验收场景**：

1. **给定** 系统需要设置或清除全局故障，**当** 维护者追踪写入来源时，**则** 只能定位到正式故障入口或其明确授权的收口路径。
2. **给定** 系统需要刷新最近结果，**当** 某个查询、适配或辅助服务被执行时，**则** 这些路径不会顺手改写最近结果。
3. **给定** 系统需要受理触发、推进处理并记录时间事实，**当** 维护者检查触发队列与当前时间的写入时，**则** 可以明确谁能改、何时能改以及改后对外如何体现。

---

### 用户故事 3 - 业务服务与主控总容器解耦（优先级：P1）

作为领域逻辑开发者，我希望领域服务和用例服务只处理自己职责范围内的最小输入与输出，而不是直接拿到整个主控总上下文，这样后续新增规则时不会每次牵动大片代码。

**为什么是这个优先级**：如果服务继续依赖总容器，任何细小业务变更都会放大为跨层耦合和全链路连带修改。

**独立验证方式**：检查领域服务和用例服务的依赖输入，确认其只接收完成职责所需的数据与端口；检查其输出落点，确认最终状态写入仍由正式入口或明确拥有者完成。

**验收场景**：

1. **给定** 某领域服务只需要一段运行快照和少量端口能力，**当** 应用层调用该服务时，**则** 不需要把整个 `system_context_t` 透传给它。
2. **给定** 某用例服务完成一次业务判断，**当** 它返回结果时，**则** 结果会落到预定的状态拥有者或编排入口，而不是由该服务越级改写主控总状态。
3. **给定** 领域服务需要参与流程推进，**当** 维护者审查其职责时，**则** 看不到其越级控制主控主循环或直接改写无关运行对象。

---

### 用户故事 4 - 运行期边界由回归验证持续守住（优先级：P2）

作为后续扩展与回归验证的维护者，我希望系统具备一组围绕边界约束的回归验证，用来证明只读查询不会改状态、故障只能从正式入口进入、会话结束后状态落点唯一，这样结构约束不会在后续开发中被悄悄打回原状。

**为什么是这个优先级**：本轮改造改的是结构约束，不是新业务能力；如果没有边界型回归，后续新增功能很容易再次把状态和职责写散。

**独立验证方式**：执行围绕边界的回归场景，确认读写边界、跨层边界和最终状态落点都能稳定通过验证。

**验收场景**：

1. **给定** 某个只读查询入口被调用，**当** 查询完成后，**则** 不会产生任何关键运行状态写入副作用。
2. **给定** 非正式路径尝试注入全局故障，**当** 系统处理该操作时，**则** 该写入会被拒绝或被规范化为正式入口处理。
3. **给定** 一次会话正常结束或异常结束，**当** 维护者检查最终状态落点时，**则** 会话结果只落到唯一、可解释的位置，而不是分散在多个对象中。

### 边界场景

- 当某个运行数据只在单次服务调用中临时存在时，系统不得默认将其提升为总上下文共享状态。
- 当适配层只负责对接外部输入输出时，系统不得允许它直接改写业务状态或越级决定主控流程结果。
- 当发现适配层、查询路径或领域服务存在跨层直写时，系统必须把该路径视为实现违规，并重构为正式入口或明确拥有者写入，而不是用运行时转发继续保留原写法。
- 当不同外部适配层需要驱动主控处理时，系统必须允许它们通过各自明确的应用入口提交事实或请求；其中 CLI 正式命令入口固定为 `process_formal_command`，但任何适配层都不得直接碰调度队列。
- 当本期清理正式入口边界时，系统不得继续保留 `start_wash_cycle`、`stop_wash_cycle`、`acknowledge_fault` 一类兼容性用例入口作为第二套主路径。
- 当查询路径需要返回当前状态视图时，系统必须保持只读语义，不得顺带刷新最近结果、故障状态或会话结果。
- 当领域服务完成判断后，系统必须把最终状态写入留给正式入口或明确拥有者，而不是让领域服务直接改写主控总容器。
- 当会话状态、执行状态、等待状态、程序快照、端口适配、主循环运行数据同时存在时，系统必须明确每个对象负责什么、不负责什么，以及谁拥有最终解释权。
- 当系统需要设置全局故障时，系统必须通过正式故障入口进入，而不是由适配层、查询路径或普通领域服务直接注入。
- 当外部适配层需要驱动一次处理时，系统必须只允许其提交事实或请求，而不得直接改写 `pending_triggers` 或其他主循环内部调度状态。
- 当系统需要结束一次会话时，系统必须保证最终状态落点唯一，避免最近结果、会话结果和其他运行对象彼此冲突。
- 当一次会话结束时，系统必须把最终会话结论唯一落在 `wash_session.final_session_result`，而把最近结果保持为最近一次对外处理结论的投影。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须将 `system_context_t` 定义为运行期总容器，只承载确有共享必要且需要主控统一解释的运行事实。
- **FR-002**：系统必须为 `system_context_t` 中的每类共享状态说明其用途、共享原因和最终解释责任。
- **FR-003**：系统必须将只属于单个服务、单次处理或局部计算过程的临时数据排除在 `system_context_t` 默认边界之外。
- **FR-004**：系统必须为全局故障、最近结果、触发队列和当前时间定义少数固定的正式写入口。
- **FR-004A**：系统必须将触发队列定义为主循环内部调度机制；外部适配层只能通过正式提交入口上报事实或请求，不得直接改写 `pending_triggers` 或 `pending_trigger_count`。
- **FR-005**：系统必须明确每个关键运行状态允许谁改写、何时改写，以及改写后的对外表现。
- **FR-006**：系统必须阻止查询路径把只读访问变成关键运行状态写入。
- **FR-007**：系统必须确保适配层不能直接改写业务状态、最近结果或会话结果，只能通过正式应用入口提交事实或请求。
- **FR-007A**：系统必须将 `process_formal_command` 固定为 CLI 正式命令受理入口；除 CLI 以外的外部适配层必须通过各自明确的应用入口提交事实或请求，但不得为了入口统一而直接复用或绕过主循环调度边界。
- **FR-007B**：系统必须删除现有兼容性用例入口（如 `start_wash_cycle`、`stop_wash_cycle`、`acknowledge_fault`）作为正式受理路径的保留地位，不得继续维持第二套与正式命令入口并行的主路径。
- **FR-008**：系统必须确保领域服务和用例服务只依赖完成职责所需的最小输入，而不是默认依赖整个 `system_context_t`。
- **FR-009**：系统必须确保领域服务和用例服务的输出落到预定状态拥有者、正式入口或明确收口路径，而不是由服务自身越级写回主控总状态。
- **FR-010**：系统必须明确区分会话状态、执行状态、等待状态、程序快照、端口适配和主循环运行数据的职责边界。
- **FR-011**：系统必须为每个运行期对象定义其负责范围、不负责范围以及最终解释权归属。
- **FR-011A**：系统必须将会话最终结论唯一落在 `wash_session.final_session_result`，并将最近结果限制为最近一次正式处理对外投影，不得把两者同时定义为独立最终落点。
- **FR-012**：系统必须清理适配层直接改业务状态、查询顺手改结果、领域服务越级控制主控流程等跨层直连问题。
- **FR-012A**：系统必须将跨层直写视为实现违规；一旦发现，必须改回正式入口或明确拥有者写入，并通过回归验证防止该违规路径再次出现。
- **FR-013**：系统必须保留现有工步段模型语义、现有 JSON 工艺规则语义和现有主控外部行为稳定，不得把本次边界收紧扩大成流程模型重做。
- **FR-014**：系统必须为本次边界约束补充回归验证，至少覆盖只读查询不改状态、故障只能从正式入口进入、会话结束后状态落点唯一三类场景。
- **FR-015**：系统必须让后续新增功能在评审时能够明确判断其影响范围是否局限于少数边界对象与固定状态入口。

### 范围与简化要求（必填）

- **SS-001**：本次只收紧主控骨架的状态边界、职责边界和跨层访问边界，不增加新的洗车工艺能力。
- **SS-002**：本次不修改工步段模型语义，不重新定义现有流程模型。
- **SS-003**：本次不修改 JSON 工艺规则本身，不扩展程序在线编辑、保存或界面层能力。
- **SS-004**：本次不扩展真机驱动能力，只在现有运行骨架内收敛边界和验证约束。
- **SS-005**：本次允许删除仅为兼容旧调用方保留的第二套用例入口，但不把该删除动作扩展为对整个命令协议或 CLI 行为的重写。

### 嵌入式 C 与安全要求（适用时填写）

- **EC-001**：关键运行状态的写入口必须保持显式、可审查和可追踪，不得通过隐式共享、旁路指针或顺手赋值形成隐藏写路径。
- **EC-002**：运行期对象边界调整不得削弱现有主控的故障安全语义、状态机清晰度或异常处理一致性。
- **EC-003**：公开接口和关键边界约束必须继续具备可审查的中文注释说明，便于后续扩展时识别允许依赖和禁止依赖。

### 关键实体（若功能涉及数据则必填）

- **总上下文**：即 `system_context_t`，表示主控运行期总容器，只包含需要跨对象共享和统一解释的事实。
- **关键运行状态**：指全局故障、最近结果、触发队列、当前时间，以及会话结束后需要统一解释的最终状态结果。
- **会话最终结果**：指一次会话结束后的唯一正式业务结论，固定落在 `wash_session.final_session_result`。
- **运行期对象**：指会话状态、执行状态、等待状态、程序快照、端口适配和主循环运行数据等需要明确边界的对象。
- **正式写入口**：指被允许改写关键运行状态的固定入口或拥有者。
- **触发提交入口**：指适配层向主控提交外部事实或请求的正式入口；它不等同于允许适配层直接操作触发队列。
- **CLI 正式命令入口**：指 `process_formal_command` 对 CLI 协议暴露的正式受理入口；它不自动代表所有外部适配层必须复用同一接口。
- **兼容性用例入口**：指 `start_wash_cycle`、`stop_wash_cycle`、`acknowledge_fault` 一类历史包装入口；本期将其从正式主路径中移除。
- **边界型回归验证**：指用于证明边界仍被守住的验证场景，而非用于扩展业务工艺能力的流程验证。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：在本次纳入范围的关键运行状态中，100% 都能定位到固定正式写入口，且不存在来源不明的并行写入口。
- **SC-002**：在本次纳入范围的领域服务和用例服务中，100% 不再默认依赖整个 `system_context_t` 作为输入。
- **SC-003**：围绕只读查询的回归验证中，关键运行状态非预期写入数量为 0。
- **SC-004**：围绕全局故障入口的回归验证中，非正式路径直接注入故障的成功次数为 0。
- **SC-005**：围绕会话结束的回归验证中，最终状态落点冲突数量为 0。
- **SC-006**：后续新增单项功能在评审时，都能用明确边界说明其影响对象和固定状态入口，不再默认扩大到整个主控总容器。
- **SC-007**：本次纳入范围内已识别的跨层直写路径清理完成率达到 100%，且新增回归验证后重复出现数量为 0。
- **SC-008**：本次纳入范围内保留并行兼容性用例入口的数量为 0，正式命令入口不再存在第二套等价主路径。

## 验证方式（必填）

- **V-001**：逐项审查 `system_context_t` 与相关运行对象，验证共享状态、临时数据和最终解释权的边界是否清晰。
- **V-002**：逐类追踪全局故障、最近结果、触发队列和当前时间的写入路径，验证是否只通过正式入口改写。
- **V-002A**：验证适配层只能通过正式提交入口驱动处理，不能直接改写 `pending_triggers`、`pending_trigger_count` 或其他主循环内部调度状态。
- **V-002B**：验证 CLI 路径通过 `process_formal_command` 受理正式命令，而其他外部适配层通过各自明确的应用入口提交事实或请求，且两类路径都不直接触碰调度队列。
- **V-002C**：验证 `start_wash_cycle`、`stop_wash_cycle`、`acknowledge_fault` 一类兼容性用例入口已从正式主路径中移除，不再作为并行受理入口参与回归口径。
- **V-003**：检查领域服务和用例服务的输入输出，验证其依赖是否收缩为最小必要集合，且输出落点明确。
- **V-004**：审查运行期对象职责说明，验证会话状态、执行状态、等待状态、程序快照、端口适配和主循环运行数据之间不存在职责重叠或最终解释权冲突。
- **V-005**：执行边界型回归验证，覆盖只读查询不改状态、故障只能从正式入口进入、会话结束后状态落点唯一三类场景。
