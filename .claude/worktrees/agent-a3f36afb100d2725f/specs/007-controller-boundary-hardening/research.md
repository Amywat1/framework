# 研究记录：主控骨架边界收紧

## 决策 1：继续保持 `C17 + Linux + CMake/CTest + JSON + 模拟驱动`

**Decision**：本期不新增技术栈，继续沿用现有 C17、Linux、CMake/CTest、JSON 程序配置和模拟驱动体系。  
**Rationale**：
- 本期目标是边界收紧，不是能力扩张；引入新栈只会扩大不必要变更面。
- 当前主控、仿真驱动和测试目录已经稳定运行在该栈上，迁移工具不会直接解决状态边界失控问题。
- 保持现有栈有利于把关注点集中在组合根、状态拥有者和回归验证。  
**Alternatives considered**：
- 引入新的测试框架或运行时：收益不足，且违背用户明确给出的技术栈约束。
- 引入消息总线或脚本层：会把边界问题换成新基础设施问题。

## 决策 2：保留 `system_context_t`，但把它固定为应用层组合根

**Decision**：保留 `system_context_t`，将其定义为应用层组合根，只承载共享运行事实、端口、计数器和少量统一解释所需的状态。  
**Rationale**：
- 当前代码已经围绕 `system_context_t` 形成统一主控外壳，直接删除会把本期变成架构重写。
- 规格明确本期重点是“保留但收口”，不是“移除总上下文”。
- 组合根定位允许应用层统一装配对象，同时阻止领域层把总上下文当作万能依赖。  
**Alternatives considered**：
- 完全移除总上下文：改动过大，且不符合本期范围。
- 维持现状不定义边界：后续功能仍会继续向总容器塞状态。

## 决策 3：领域层继续只接收最小依赖切片，不直接依赖组合根

**Decision**：继续使用并强化现有 `wash_session_service_args_t`、`wash_execution_service_args_t`、`program_snapshot_service_args_t` 等参数切片模式，禁止新的领域服务直接接收 `system_context_t`。  
**Rationale**：
- 现有领域服务头文件已经证明最小依赖切片模式可行，说明无需新增抽象即可落地。
- 组合根与领域规则之间保持切片边界，能够显著降低新增功能的波及面。
- 服务参数切片天然适合做“允许依赖什么、不允许依赖什么”的契约。  
**Alternatives considered**：
- 让领域服务继续接收 `system_context_t`：会把应用编排、端口、状态结果和领域规则混在一起。
- 为每个服务再新建一层容器工厂：复杂度过高，超出当前需求。

## 决策 4：关键运行状态必须收口到固定写入口

**Decision**：把关键运行状态按拥有者固定如下：`global_fault_*` 由 `process_wash_trigger_execute()` 改写；最近结果与迁移记录由 `runtime_event_recorder_*()` 落点；`pending_triggers` 与 `pending_trigger_count`、`current_time_ms` 由 `main_loop_*()` 维护。  
**Rationale**：
- 当前代码中这些写路径已经部分集中，适合在本期固化为正式规则。
- 关键状态的核心问题不是能不能写，而是有没有唯一、可审查、可回归验证的写入口。
- 明确入口后，查询、适配和领域服务就能自然退回只读或事实提供者角色。  
**Alternatives considered**：
- 只在代码注释里口头约束：不足以形成长期边界。
- 允许多个用例按需写入最近结果或故障：会再次制造解释不一致。

## 决策 5：主循环只拥有时间与触发队列，不拥有业务最终解释权

**Decision**：把 `main_loop_*()` 的职责限制为推进时间、维护触发队列、挑选待处理事件并调用正式入口，不直接承担业务最终状态解释。  
**Rationale**：
- `current_time_ms` 和 `pending_triggers` 本质上是运行时调度事实，适合留在主循环边界内。
- 一旦主循环开始直接决定最近结果、会话结论或故障解释，组合根边界会再次混乱。
- 让主循环只负责“调度”和“时钟”，更符合显式状态机外壳定位。  
**Alternatives considered**：
- 把业务结果判断继续留在主循环：会使 `main_loop.c` 再次变成逻辑杂糅点。
- 把时间写入分散到各用例：会损害时间语义的一致性。

## 决策 6：只读查询保持纯投影，不得产生写副作用

**Decision**：`query_wash_session_status_execute()` 继续作为只读状态视图入口，只允许从 `system_context_t` 和已存在运行对象中投影视图，不得刷新最近结果、故障状态或会话状态。  
**Rationale**：
- 规格明确要求“只读查询不改状态”必须成为回归验证的一部分。
- 当前头文件和实现已经具备只读语义注释，适合直接升级为正式契约。
- 查询纯投影有利于将来扩展更多状态视图而不污染业务结果。  
**Alternatives considered**：
- 查询时顺手刷新最近结果：短期看似方便，长期会让结果语义失真。
- 把查询并入触发队列：会混淆只读访问和业务事件处理。

## 决策 7：运行期对象按拥有者划分最终解释权

**Decision**：将运行期对象边界固定为：`wash_session_t` 解释会话级结论，`wash_execution_t` 解释当前工步执行态，`wait_condition_t` 解释等待与超时，`program_snapshot_t` 解释会话绑定程序快照，`wash_session_status_view_t` 只解释只读视图，不反向拥有业务状态。  
**Rationale**：
- 当前对象已经存在，但边界若不明，后续代码仍可能跨对象直接补字段和补写路径。
- “谁拥有最终解释权”是防止会话结果、执行结果和最近结果互相打架的关键。
- 对象边界清晰后，状态落点唯一性才可验证。  
**Alternatives considered**：
- 继续让多个对象重复持有相同结论：会增加冲突和同步成本。
- 让状态视图反向驱动业务状态：违反只读投影原则。

## 决策 8：边界型回归验证按契约、集成、单元三层覆盖

**Decision**：本期验证重点不是工艺流程，而是边界约束；采用契约测试验证接口边界，集成测试验证正式入口与状态落点，单元测试验证对象拥有权与辅助工具行为。  
**Rationale**：
- 规格明确本期改的是结构约束，不是业务能力。
- 仅靠代码评审无法长期防止边界回退，必须把关键约束转成可重复执行的测试。
- 三层测试可以分别覆盖“接口承诺”“跨对象编排”“局部状态工具”三个风险面。  
**Alternatives considered**：
- 只做人工评审：无法形成稳定回归。
- 只做流程仿真：看得到流程能跑，看不到边界是否被破坏。
