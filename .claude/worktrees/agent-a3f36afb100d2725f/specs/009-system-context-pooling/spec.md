# 功能规格：system_context 组合根生命周期收口与静态池化
**功能分支**：`009-system-context-pooling`  
**创建日期**：2026-05-09  
**状态**：草稿  
**输入**：用户要求将 `system_context_t` 收敛为对外不透明、由内部静态对象池统一提供的组合根运行时资源，不改变洗车业务流程语义。

## Clarifications

### Session 2026-05-09

- Q: 释放后的旧句柄在槽位再次分配后是否仍必须可检测并报错？ → A: 只保证检测“未占用/已释放”句柄；槽位再次分配后的旧指针复用视为调用方越界，不要求系统提供跨复用周期识别。

## 澄清、假设与权衡（必填）

- **假设**：当前系统在单进程内受控运行，`system_context_t` 的正式实例数量很少，适合由内部固定容量静态对象池承载；主程序、调度器、适配层和测试会在同一特性内迁移到新生命周期入口；必要白盒验证仍可存在，但只能验证正式资源边界，不能恢复外部自带存储模型。
- **未决问题**：不适用。
- **解释选择**：本规格将“对外不透明”解释为外部只能通过公开生命周期接口获取和释放 `system_context_t` 句柄，不能依赖结构体布局、字段可见性或外部自带存储；将“组合根资源”解释为主控运行时对外暴露的统一资源边界，而不是公共状态容器；将“重置”解释为不释放占用权前提下清空运行态重新开始；将“释放”解释为结束占用并归还内部池资源。本次明确要求正式实例唯一来源为**内部静态对象池**，不再把“堆/静态/其他方式均可”留作开放项。
- **不在范围内**：不调整洗车业务流程；不修改工步段模型；不扩展调度器业务能力；不新增硬件能力；不重构领域规则；不把 `system_context_t` 扩展成新的通用上下文容器。

## 用户场景与测试（必填）

### 用户故事 1 - 正式实例只能从统一入口获取（优先级：P1）

作为主程序和集成入口维护者，我希望所有 `system_context_t` 实例都只能通过统一正式入口获取，这样外部模块不再决定对象存储来源，生命周期拥有权也不会继续分散。

**为什么是这个优先级**：只要外部还能自己声明对象存储并交给初始化函数，生命周期边界就仍可被旧入口绕开。

**独立验证方式**：检查主程序、调度器、适配层和测试中的实例创建路径，确认它们都改为先申请正式句柄再装配使用，且外部不再自行准备 `system_context_t` 存储。

**验收场景**

1. **给定** 内部对象池当前有可用实例，**当** 主程序启动主控运行时，**则** 它通过正式获取入口拿到一个可用句柄，并在后续使用期间保持占用。
2. **给定** 适配层或测试辅助需要使用主控运行时，**当** 它请求实例时，**则** 它拿到的也是同一套正式句柄语义，而不是调用方自带对象存储。
3. **给定** 外部模块只持有公开头文件和正式接口，**当** 它集成主控运行时，**则** 它既不需要也不能依赖 `system_context_t` 的内部布局。

---

### 用户故事 2 - 重置与释放必须明确分离（优先级：P1）

作为运行时拥有者，我希望“清空运行态重新开始”和“结束生命周期归还资源”是两个不同动作，这样主控、调度器和测试可以在不混淆资源占用关系的前提下重新启动一轮运行。

**为什么是这个优先级**：如果重置和释放混在一起，调用方就无法稳定区分“重新开始一次运行”和“把对象交还系统”。

**独立验证方式**：对同一实例分别执行重置和释放，检查重置后句柄仍归当前持有者占用并可继续使用，释放后实例回到池中且旧句柄不再可用于正式运行。

**验收场景**

1. **给定** 持有者正在使用一个运行时实例，**当** 它执行重置，**则** 运行态被清空并重新开始，但该实例仍归当前持有者占用。
2. **给定** 持有者结束一次运行时生命周期，**当** 它执行释放，**则** 该实例被归还给内部对象池，旧句柄不再代表可操作的正式运行时。
3. **给定** 一个实例已经被释放，**当** 其他调用方再次获取资源时，**则** 资源提供方能够重新分配一个正式实例，并以干净运行态开始下一次使用。

---

### 用户故事 3 - 池耗尽和非法使用有稳定语义（优先级：P1）

作为调度器、测试和适配层维护者，我希望对象池耗尽、重复释放、非法句柄和释放后误用都有正式且一致的对外行为，这样外围模块可以稳定依赖边界，而不是依赖隐含假设自保。

**为什么是这个优先级**：生命周期边界的主要风险不在正常路径，而在异常路径；异常语义不清会把系统重新拉回旧模型。

**独立验证方式**：分别构造池已满、重复释放、非法句柄、释放后再次使用和跨持有者误用场景，确认返回结果、资源状态和可恢复行为符合统一定义。

**验收场景**

1. **给定** 当前没有可分配的正式实例，**当** 新调用方继续请求实例时，**则** 系统返回明确的资源不可用结果，且不影响已占用实例状态。
2. **给定** 某个实例已经被释放，**当** 相同句柄再次被释放时，**则** 系统返回明确的非法释放结果，而不会重复归还或污染内部资源状态。
3. **给定** 调用方持有一个仍可判定为未占用或已释放的失效句柄，**当** 它继续执行正式运行时操作时，**则** 系统返回明确失败结果，且不会误操作其他仍在使用中的实例。

---

### 用户故事 4 - 白盒验证只验证边界，不恢复旧模型（优先级：P2）

作为测试维护者，我希望保留必要的白盒验证能力，但这些测试只验证资源边界、拥有权边界和正式行为，而不是继续依赖外部自带对象存储或直接假设对象布局。

**为什么是这个优先级**：如果测试继续把外部自带存储当作默认模式，后续每次收紧边界都会被回归测试反向拉回旧语义。

**独立验证方式**：检查测试辅助和白盒测试的使用方式，确认它们通过正式生命周期入口构造实例；必要内部断言只围绕正式生命周期状态、资源复用和异常语义展开。

**验收场景**

1. **给定** 白盒测试需要观察正式生命周期的占用与归还状态，**当** 它执行验证时，**则** 它使用受控测试入口验证正式资源语义，而不是外部构造对象存储。
2. **给定** 回归测试覆盖主程序、适配层和调度器路径，**当** 它们创建运行时实例时，**则** 全部走与生产路径一致的正式生命周期。

## 边界场景

- 当当前没有可分配的正式实例时，新的获取请求必须返回明确失败结果，且不得影响已占用实例。
- 当句柄为空、非法、已释放或来自非正式来源时，所有正式生命周期与运行时操作都必须返回明确失败结果。
- 当实例被重置后再次使用时，系统必须表现为一次新的运行开始，而不是延续上一次残留状态。
- 当实例被释放后再次被获取时，新持有者看到的必须是干净、可重新装配的运行态。
- 当调用方重复释放同一实例时，系统必须拒绝该动作，并保持内部资源管理状态不变。
- 当旧句柄仍可判定为未占用或已释放状态时，任何继续使用都必须返回明确失败；若对应槽位已被重新分配，则旧指针复用视为调用方边界违规，不要求跨复用周期识别。
- 当主程序、调度器、适配层或测试仍试图沿用旧的创建、初始化、销毁方式时，系统必须提供清晰迁移边界，不保留长期并存的双轨入口。
- 当测试需要观察内部生命周期语义时，白盒能力只能暴露验证所需的受控边界，不能恢复对外可构造、可持有对象存储的旧模型。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须提供一套正式的 `system_context` 生命周期接口，用于获取、使用、重置和释放运行时实例。
- **FR-002**：系统必须禁止外部模块、主程序和测试通过自带对象存储直接构造正式 `system_context` 实例。
- **FR-003**：系统必须将所有正式 `system_context` 实例的来源收敛为**内部静态对象池**，不再允许堆分配、外部自带存储或其他并行来源参与正式构造。
- **FR-004**：系统必须把对象池定义为**固定容量、无运行期扩容**的内部资源；当无可用实例时，获取接口必须返回明确的资源不可用结果。
- **FR-005**：系统必须为每个正式实例定义完整的生命周期状态规则，至少覆盖可获取、已占用、占用中可重置、已释放失效和释放后可再次被正式获取这些状态语义。
- **FR-006**：系统必须保证一个池内实例在被占用期间只归当前持有者使用，不能被第二个调用方重复获取。
- **FR-007**：系统必须将“重置运行态”和“释放资源”定义为两个不同动作，且二者的对外行为、后续可用性和拥有权变化必须明确区分。
- **FR-008**：系统必须在重置时清空上一次运行遗留的运行态，使同一持有者能够以干净状态重新开始，而不丢失该实例的占用关系。
- **FR-009**：系统必须在释放时结束该实例的占用关系、归还静态池资源，并使旧句柄不再代表一个可操作的正式运行时。
- **FR-010**：系统必须保证被释放后重新分配的实例以干净运行态重新开始，不能泄漏上一位持有者的运行结果、待处理状态或边界标记。
- **FR-011**：系统必须为生命周期异常场景定义正式行为，至少覆盖池耗尽、重复释放、非法句柄、释放后再次使用（当句柄仍可判定为未占用或已释放时）和无效重置。
- **FR-012**：系统必须保证异常场景的失败结果不会污染其他仍在使用中的实例，也不会破坏内部对象池管理状态的可恢复性。
- **FR-012A**：系统只要求对当前可判定为未占用、已释放或非正式来源的句柄返回显式错误；如果槽位已被重新分配，旧指针复用视为调用方边界违规，不承诺跨复用周期识别。
- **FR-013**：系统必须让主程序、调度器、测试辅助和适配层统一使用同一套正式生命周期接口，不得保留并行的旧式入口。
- **FR-014**：系统必须更新现有依赖 `system_context` 生命周期的使用路径，使外围模块不再依赖“调用方拥有对象存储”的旧假设。
- **FR-015**：系统必须保留必要的白盒验证能力，但这些能力只能用于验证正式生命周期语义、拥有者边界和正式行为，不能恢复外部直接构造对象存储的使用方式。
- **FR-016**：系统必须把 `system_context` 保持为主控运行时的组合根资源，不得借本次改造继续向其中下沉新的业务职责、临时状态集合或额外平台能力。
- **FR-017**：系统必须在公开边界上维持 `system_context` 的不透明语义，使外部调用方不需要也不能依赖其内部结构才能完成正式集成。
- **FR-018**：系统必须为本次改造补齐回归验证，覆盖正常获取与释放、重置后继续运行、池耗尽、实例复用、重复释放、非法句柄和释放后误用等场景。

### 范围与简化要求（必填）

- **SS-001**：本次只收敛 `system_context` 生命周期、资源拥有权、内部静态对象池语义和外部统一入口，不扩展业务能力。
- **SS-002**：本次不调整洗车业务流程、工步段模型、领域规则和调度器业务策略，只改造它们获取与持有运行时资源的方式。
- **SS-003**：本次不新增硬件能力、远程控制能力、管理界面或新的通用上下文抽象。
- **SS-004**：本次对测试的调整只服务于边界收紧和语义回归，不把白盒便利性放在正式生命周期边界之前。
- **SS-005**：本次必须把旧生命周期模型的残留使用路径清理到正式迁移边界内，不能让“新的正式生命周期语义”和“旧的外部自带存储语义”长期并存。

### 嵌入式 C 与安全要求（适用时填写）

- **EC-001**：所有资源获取、重置、释放和异常返回都必须具备显式、可审计的成功/失败语义，不得依赖隐式拥有权转移。
- **EC-002**：静态对象池的容量、资源不可用语义和实例复用行为必须保持有界、可预测，不得引入运行时扩容。
- **EC-003**：公开边界必须保持不透明类型语义，外部接口不得要求调用方了解内部字段布局才能正确使用。
- **EC-004**：非法句柄、重复释放和释放后误用必须返回明确错误结果，并保持其他运行时实例和内部资源管理状态安全。
- **EC-005**：测试辅助、适配层和调度器不得通过旁路入口绕过正式生命周期，否则视为边界违规。
- **EC-006**：与 `system_context` 相关的公开接口说明和测试契约必须使用清晰中文描述拥有权、重置和释放约束，避免再次出现混合语义。

### 关键实体（若功能涉及数据则必填）

- **system_context 运行时句柄**：表示调用方通过正式生命周期入口获得的主控运行时实例引用，具有明确的占用、重置、释放和失效语义。
- **system_context 静态对象池**：表示内部固定容量、统一管理正式实例来源的资源边界，负责实例获取、占用、释放、失效和再次可获取的正式语义。
- **运行时持有者**：表示当前合法占用某个 `system_context` 实例的外部使用方，例如主程序、调度器或测试路径；其职责是按正式语义使用、重置并释放实例。
- **运行态**：表示一次主控运行过程中产生的可清空状态集合；它可以被重置以重新开始，但不等同于资源本身的拥有权。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：100% 的主程序、调度器、测试辅助和适配层运行时创建路径都改为通过正式生命周期入口获取实例，不再依赖外部自带对象存储。
- **SC-002**：围绕池耗尽、重复释放、非法句柄和释放后误用的正式回归场景 100% 产生文档化且一致的结果。
- **SC-003**：在“获取 -> 使用 -> 重置 -> 继续使用 -> 释放 -> 再次获取”的完整回归路径中，100% 能验证重置与释放语义清晰分离，且实例复用后无残留状态泄漏。
- **SC-004**：公开边界上的 `system_context` 使用方式在 100% 的集成路径中不再要求外部知道内部结构或对象存储方式。
- **SC-005**：本次改造后的生命周期回归验证 100% 覆盖正式正常路径与异常路径，且不存在继续依赖旧生命周期模型才能通过的测试。

## 验证方式（必填）

- **V-001**：通过主程序或等效组合根路径验证正式实例获取、装配使用和释放行为。
- **V-002**：通过同一实例的重置后继续运行场景验证“重置不释放、释放才归还”的正式边界。
- **V-003**：通过池耗尽与释放后复用场景验证静态对象池不可扩容、资源归还和再次获取语义。
- **V-004**：通过重复释放、非法句柄和释放后误用场景验证异常结果与内部资源管理安全性；其中释放后误用只要求覆盖仍可判定为未占用或已释放的句柄。
- **V-005**：通过测试辅助、调度器和适配层迁移验证它们已统一走正式生命周期入口。
- **V-006**：通过白盒回归验证确认必要内部观察能力仍可用，但不再允许外部构造正式对象存储。

## 依赖与默认前提

- 现有主控运行时相关模块会在本特性中同步迁移到正式生命周期入口，不保留长期双轨模式。
- 内部正式实例来源固定为静态对象池，但池容量是私有实现常量，不作为公开 API 的一部分。
- 必要的内部测试观察能力可以通过受控边界保留，但这些边界不会重新变成公开构造入口。
- 本期允许对主程序、调度器、测试辅助、适配层和相关契约文档做协同调整，以完成生命周期语义收口。
