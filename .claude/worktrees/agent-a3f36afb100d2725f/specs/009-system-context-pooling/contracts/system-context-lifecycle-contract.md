# Contract：`system_context` 正式生命周期

## 目的

定义 `system_context_t` 作为正式组合根资源时，对外唯一允许的获取、重置、释放与异常语义。

## 参与方

- 生命周期调用方：`main`、调度器创建路径、测试辅助、适配层装配路径
- 正式生命周期接口：`system_context` 公开头文件中的 acquire / reset / release / 装配接口
- 内部资源管理实现：私有静态预分配对象池、空闲链表头管理、槽位状态校验

## 契约规则

1. `system_context_t` 只能通过正式 acquire 入口获取，不能由调用方在栈上、静态区或堆上自带存储后再初始化。
2. acquire 成功时返回一个当前已占用的正式句柄；失败时返回显式资源不可用或非法请求结果。
3. reset 只允许作用于当前已占用实例；成功后运行态被清空，但端口装配关系继续保留。
4. release 只允许成功一次；成功后句柄失去正式拥有权，实例回到内部资源管理体系。
5. 对空句柄、非正式来源句柄、未占用实例执行 reset / release / 运行态操作，必须返回显式错误或失败结果。
6. release 之后继续缓存并复用旧 `system_context_t *` 属于调用方边界违规；实现至少要对可检测的已释放场景给出失败结果。
7. 内部分配策略固定为：空闲链表头提供下一个可分配实例，release 总是把实例挂回空闲链表头；该策略作为当前实现契约，不向公开 API 泄漏。
8. `in_use` 只用于判断句柄是否处于合法占用状态，不承担“从哪些节点中找可分配对象”的职责。

## 可验证结果

- 主程序不再调用 `system_context_initialize()` 或依赖 `system_context_create()` 的堆语义。
- 生命周期边界测试覆盖 acquire、reset、release、重复 release、资源不可用和释放后误用。
- reset 与 release 的结果可以通过测试稳定区分：前者保留装配，后者结束占用。
- 白盒实现验证能够确认空闲链表始终按头弹出/头插回收运行。
