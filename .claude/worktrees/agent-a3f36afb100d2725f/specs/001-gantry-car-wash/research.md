# 研究记录：往复式龙门洗车机控制软件

## 决策 1：采用 Linux 用户态 C17 主控守护进程

**Decision**：使用 Linux 用户态单进程守护进程作为主控程序，语言采用 C17。

**Rationale**：
- 用户已明确指定 C 语言与 Linux 平台。
- 单进程守护进程更容易控制时序、日志、配置和故障恢复边界。
- 在本项目范围内，Linux 提供的 `epoll`、`timerfd`、线程与文件 I/O 已足够覆盖控制、事件驱动和配置持久化需要。

**Alternatives considered**：
- 多进程架构：隔离性更强，但会引入 IPC、恢复一致性和部署复杂度。
- RTOS/裸机方案：与用户指定平台不符。

## 决策 2：采用“DDD 思想 + 六边形架构”的裁剪实现

**Decision**：在 C 语言环境中采用裁剪版 DDD 与六边形架构：领域层聚焦规则和状态机，应用层编排用例，适配器层承接硬件、配置、日志和操作接口。

**Rationale**：
- DDD 适合承载洗车程序、故障分级、状态转移等高业务含义规则。
- 六边形边界能把硬件 I/O、操作接口和持久化从核心规则中隔离出来，便于仿真、单元测试和 HIL。
- 裁剪版实现避免把 Java/C# 风格框架概念生搬到 C 中，只保留边界与职责，不引入复杂容器或运行时反射。

**Alternatives considered**：
- 三层结构：对简单 CRUD 足够，但对高频状态机与多适配器控制场景表达力不够。
- 纯事件脚本化状态机：简单但难以承接程序配置、领域对象和测试隔离需求。

## 决策 3：核心控制采用显式分层状态机

**Decision**：至少为洗车周期、龙门运动、刷组控制、药剂投加和故障恢复建立显式状态机。

**Rationale**：
- 项目宪章明确要求关键控制建议使用显式状态机。
- 洗车机具有强顺序性、并发协同和故障恢复需求，状态机比隐式标志位更容易验证与审计。
- 故障分级、回到安全待机和重新预检天然适合用状态转移建模。

**Alternatives considered**：
- 基于标志位和 if/else 的过程控制：初期编码快，但极易在故障恢复和并发下失控。
- 表驱动脚本引擎：灵活，但超出当前最小可行范围。

## 决策 4：程序配置和运行日志使用文件持久化

**Decision**：洗车程序参数、车辆类别映射和系统参数使用版本化配置文件持久化；运行事件和故障记录采用追加式日志文件。

**Rationale**：
- 单机嵌入式 Linux 控制系统通常不需要数据库。
- 文件持久化易于备份、恢复和离线维护，也便于维护模式下导入导出。
- 追加式日志能满足追溯、告警分析和维保排障需要。

**Alternatives considered**：
- 嵌入式数据库：会增加部署和维护复杂度，当前价值不足。
- 全部只驻留内存：不满足参数留存和故障追溯需求。

## 决策 5：操作接口先定义逻辑契约，不绑定具体传输

**Decision**：在计划阶段先定义操作命令、配置修改和事件输出的逻辑契约，不把计划锁死到特定 HMI 技术栈或传输协议。

**Rationale**：
- 规格关注的是“程序选择、维护配置、异常处置”，而不是界面技术实现。
- 逻辑契约稳定后，可在后续实现中接入本地 HMI、CLI、串口终端或本地 IPC，而不影响领域与应用层。
- 这样更符合六边形架构“入站端口先于具体适配器”的原则。

**Alternatives considered**：
- 直接规定 HTTP/REST：对本地嵌入式控制系统过重。
- 不定义契约只写内部函数：会削弱任务拆解和测试边界。

## 决策 6：测试策略采用“领域单测 + 仿真集成 + HIL 验证”

**Decision**：建立三层验证：
1. 领域层单元测试验证状态机、程序规则和故障分级；
2. 集成/仿真测试验证应用编排与适配器交互；
3. HIL 验证真实硬件的时序、安全停机和异常恢复。

**Rationale**：
- 单靠 HIL 成本高，且难以覆盖大量边界。
- 单靠单元测试无法保证硬件时序和安全联锁。
- 三层组合最适合控制软件的成本与风险平衡。

**Alternatives considered**：
- 仅人工联调：回归成本高且不可重复。
- 仅软件仿真：不足以证明硬件安全行为。
