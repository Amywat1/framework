# 实施计划：往复式龙门洗车机控制软件

## 概要

构建一套运行在 Linux 平台上的往复式龙门洗车机主控软件，负责在单车位场景下协调龙门往复运动、顶刷/侧刷、多路药剂、安全联锁和异常处置。实现上采用“DDD 领域模型 + 六边形架构适配器边界 + 显式状态机”的组合：领域层负责洗车程序、洗车周期、故障分级和状态机规则；应用层负责编排洗车用例；适配器层对接 HMI/维护接口、硬件 I/O、配置存储和事件日志。

## 技术上下文

**语言/版本**：C17  
**主要依赖**：POSIX 线程/互斥量/条件变量、`timerfd`、`eventfd`、`epoll`、标准文件 I/O、可替换的 Linux 硬件访问适配层  
**存储**：版本化配置文件、程序参数文件、追加式事件/故障日志文件  
**测试方式**：领域单元测试、应用层集成测试、适配器仿真测试、HIL 联调测试  
**目标平台**：Linux 工控平台  
**项目类型**：嵌入式控制守护进程  
**性能目标**：当前阶段以流程闭环、状态机正确性和安全动作有效性为主，不将主控制循环周期、安全停机时延和预检时长作为阻塞验收指标；相关时序数据仅记录观测结果，后续阶段再收敛量化指标  
**约束**：单机单车位；断电/急停后必须回到安全待机；禁止断点续洗；控制逻辑必须显式状态机；保持 C 语言下可维护的 DDD/六边形分层，不引入不必要运行时框架  
**规模/范围**：1 套主控守护进程、1 套维护/操作接口、1 条龙门运动轴、顶刷/侧刷执行链路、4-8 路药剂通道、若干安全联锁与传感输入

## 假设、问题与权衡

**假设**：硬件执行器和传感器已有稳定电气接口；Linux 平台已具备可靠的 GPIO/串口/CAN/Modbus 等底层访问能力中的至少一种；操作员/HMI 与主控位于同一设备或同一局域控制单元。  
**未决问题**：无  
**被拒绝的方案**：  
1. 将系统设计为纯过程式单文件状态机：被拒绝，因为用户明确要求 DDD 与六边形，且后续程序配置、故障分级和维护模式需要清晰边界。  
2. 设计为多进程或服务网格：被拒绝，因为当前单机控制范围有限，多进程只会增加时序和故障复杂度。  
3. 采用断点续洗恢复：被拒绝，因为已澄清恢复策略必须统一回到安全待机并从起点重新开始。  
**最小可行方案**：单个 Linux 主控守护进程，内部按领域层/应用层/适配器层组织代码；通过显式状态机驱动洗车周期；以本地操作接口接受程序选择和维护命令；以配置文件加载程序定义；以日志文件记录事件。  
**范围边界**：当前只规划单机主控软件；不规划支付、门店云平台、远程 OTA、吹干联动、复杂多工位协同或视觉识别算法。

## 嵌入式 C 安全计划

**是否适用**：是  
**命名与分层**：文件/函数/变量用 `snake_case`；宏全大写；类型以 `_t` 结尾；全局变量前缀 `g_`；静态变量前缀 `s_`；头文件只放声明；领域层、应用层、适配器层严格分层。  
**格式与常量**：统一 4 空格缩进；禁用 Tab；所有控制语句带花括号；所有状态、错误码、超时、重试次数都使用具名常量或枚举。  
**错误策略**：硬件访问、程序加载、日志持久化、通信收发均返回显式错误码；安全类故障立即停机；工艺/资源类故障执行降级、跳过或安全结束。  
**等待策略**：传感器确认、运动到位、刷组就绪、药剂反馈、停止确认全部带超时和最大重试次数。  
**中断策略**：若存在中断输入，仅做事件置位/计数与轻量唤醒；不做日志、复杂判定和动态内存分配。  
**并发与内存安全**：共享状态以事件队列和受控锁保护；命令缓冲区、日志缓冲区和配置解析缓冲区都进行边界检查；所有外部指针使用前判空。  
**状态机**：洗车周期、龙门运动、刷组控制、药剂投加、故障恢复均采用显式状态机；定义上电默认状态、故障安全状态、事件日志和参数校验。  
**注释风格**：中文注释；公开头文件、端口接口、状态机转移、回调和硬件适配接口使用 Doxygen 风格。

## 复杂度跟踪
